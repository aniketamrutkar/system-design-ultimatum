System overview

Why three separate paths?
Each path has fundamentally different latency and consistency requirements. Budget enforcement needs sub-10ms with eventual consistency (a few cents of overspend is tolerable). Fraud detection can tolerate seconds of delay but needs windowed state. Reporting needs exact counts but can wait 24 hours. Coupling them forces the slowest requirement on the fastest path.
K
Kafka partitioning strategy
Partition by advertiser_id ensures all clicks for one advertiser land on the same partition. The budget enforcer needs no cross-partition coordination — just a local key lookup. ~500 partitions across ~50 brokers handle 1M msg/s.
R
Redis as the speed layer
A single Redis DECRBY is atomic and returns the new value in ~0.1ms. Shard by advertiser_id % N so no single node sees more than ~50K ops/sec. Budget stored in micros (int64) to avoid floating-point drift.
P
Parquet for cold storage
Columnar format compresses 86B daily events from ~100TB raw to ~10TB. BigQuery reads only the columns needed for each query. Partitioned by date, clustered by advertiser_id for fast rollups.
H
HyperLogLog for cardinality
For IP-level fraud detection (Rule 2), exact unique counts are too expensive. HyperLogLog gives ~0.81% error rate using only 12KB per counter. Good enough to catch bot farms without blowing memory budgets.

Hot path — budget enforcement

The most latency-sensitive component. Every click must atomically decrement the advertiser's remaining budget and, if it hits zero, kill ad serving — all within 10ms p99.

Click processing flow (click to expand)
1
Kafka consumer pulls click event
1 consumer per partition. Partitioned by advertiser_id, so each instance owns a slice of advertisers.
Each Go service instance runs a consumer group member. Since events are partitioned by advertiser_id, all clicks for one advertiser are processed by the same instance — no cross-instance coordination needed. We use manual offset commits (not auto-commit) to ensure we don't lose events on crash.
2
Blocklist pre-check (local cache)
Skip Redis round trip entirely for already-exhausted advertisers.
An in-memory bloom filter + hash set of blocked advertiser_ids. Updated via Redis pub/sub whenever any enforcer instance triggers a STOP. This saves ~99.9% of Redis calls for popular but exhausted campaigns. The cache is eventually consistent — a few extra clicks might slip through, but the Lua script catches them anyway.
3
Redis Lua script — atomic decrement
DECRBY + threshold check in a single round trip. Returns remaining budget + action code.
The Lua script runs atomically on the Redis server. It decrements the budget, checks against the threshold, and returns both the new balance and an action code (0=continue, 1=throttle, 2=stop). Idempotency is enforced by checking a small set of recently processed click_ids (SISMEMBER) before decrementing — prevents double-counting on reprocessing after crash.
4
Branch: continue / throttle / stop
Three-state response determines what happens to ad serving for this advertiser.
Code 0 (continue): ACK the Kafka offset, ads keep serving normally. Code 1 (throttle): advertiser has hit ~95% of budget — probabilistically drop 90% of ad serves to spread remaining budget across the day. Code 2 (hard stop): budget exhausted, broadcast STOP via pub/sub to all ad servers, update local blocklist.
5
Commit Kafka offset
Only after Redis confirms the decrement. At-least-once with idempotent dedup.
We commit offsets after the Redis call succeeds. If the service crashes between Redis DECRBY and Kafka commit, we'll re-process the click on restart — but the Lua script's SISMEMBER check will detect the duplicate click_id and skip the decrement. This gives us effectively exactly-once budget enforcement.
budget_decrement.lua
-- KEYS[1] = "budget:{adv_id}" -- KEYS[2] = "recent:{partition_id}" -- ARGV[1] = click cost in micros -- ARGV[2] = stop threshold (5% of daily budget) -- ARGV[3] = click_id (for idempotency) -- Idempotency check if redis.call('SISMEMBER', KEYS[2], ARGV[3]) == 1 then local cur = redis.call('GET', KEYS[1]) return {tonumber(cur), 0} -- already processed end -- Track this click_id (5min TTL for dedup window) redis.call('SADD', KEYS[2], ARGV[3]) redis.call('EXPIRE', KEYS[2], 300) -- Atomic decrement local remaining = redis.call('DECRBY', KEYS[1], ARGV[1]) if remaining <= 0 then return {remaining, 2} -- HARD STOP elseif remaining <= tonumber(ARGV[2]) then return {remaining, 1} -- THROTTLE else return {remaining, 0} -- CONTINUE end
Return code 0
Continue
Budget healthy. ACK offset, keep serving ads. This is ~99.99% of all clicks.
Return code 1
Throttle
≤5% budget remaining. Probabilistically drop 90% of ad serves to pace spending through end of day.
Return code 2
Hard stop
Budget exhausted. Broadcast STOP via pub/sub to all ad servers. Update local blocklist immediately.
The overspend problem
Between the click arriving and the STOP propagating, you might serve a few more ads. At 1M clicks/sec with ~50ms propagation delay, that's ~50K extra clicks worst case for a single viral campaign. Mitigation: set the stop threshold slightly above zero (throttle zone), and reconcile the exact overspend hourly. Advertisers accept this — a few cents of overspend is better than losing impressions.
Why Go over Flink?
Flink is great for windowed aggregations, but the budget enforcer doesn't aggregate — it does a single key lookup per event. Flink's framework overhead (checkpointing, watermarks, state backends) adds ~2-5ms of latency you don't need. A lean Go service with franz-go + go-redis gives you sub-1ms processing time per event, and you control the entire lifecycle. Each instance consumes from a fixed set of Kafka partitions with zero cross-instance coordination.
Warm path — fraud detection

Detects fraudulent clicks (same IP clicking same ad repeatedly) using sliding window aggregation on a 60-second tumbling window. Feeds a blocklist back to the hot path.

Detection rules

Rule 1 — Click stuffing: Same IP + same ad_id > 5 clicks within 60 seconds → flag all clicks from that IP/ad pair as fraudulent. Credit the advertiser back.

Rule 2 — Bot farm: Same IP clicks > 50 distinct ads within 60 seconds → block the IP entirely. Push to hot path blocklist via pub/sub.

Rule 3 — Anomalous CTR: If an ad's click-through rate exceeds 10× its campaign average, flag the ad for manual review. Don't auto-block (could be a genuinely viral creative).

Z
Redis sorted sets for windowing
Key: fraud:{ip}:{ad_id}. Each click adds a member with score = timestamp. Count clicks in last 60s via ZRANGEBYSCORE now-60 now. Old entries expire via TTL (120s) on the key. Gives exact per-window counts without unbounded state.
H
HyperLogLog for IP-level counts
For Rule 2, counting distinct ads per IP exactly would be too expensive. HyperLogLog gives ~0.81% error rate using only 12KB per counter. Key: hll:{ip}:{minute_bucket}. Sufficient to detect bot farms (50+ distinct ads) without blowing memory.
→
Feedback to hot path
When Rule 2 triggers, the fraud detector publishes the blocked IP to Redis pub/sub channel fraud:blocklist. All budget enforcer instances subscribe and update their local blocklist. Propagation: ~20-50ms.
$
Refund pipeline
Flagged clicks trigger a reverse-INCRBY on the advertiser's Redis budget counter (credit back), and mark the click as is_fraudulent=true in the Kafka topic so the cold path excludes them from billing reports.
fraud_check.lua
-- KEYS[1] = "fraud:{ip}:{ad_id}" -- ARGV[1] = click timestamp (score) -- ARGV[2] = click_id (member) -- ARGV[3] = window size in seconds (60) -- ARGV[4] = threshold (5) local now = tonumber(ARGV[1]) local window = now - tonumber(ARGV[3]) -- Add this click redis.call('ZADD', KEYS[1], now, ARGV[2]) -- Prune old entries outside window redis.call('ZREMRANGEBYSCORE', KEYS[1], 0, window) -- Count remaining (clicks in window) local count = redis.call('ZCARD', KEYS[1]) -- Set TTL to auto-expire idle keys redis.call('EXPIRE', KEYS[1], tonumber(ARGV[3]) * 2) if count > tonumber(ARGV[4]) then return {count, 1} -- FRAUD DETECTED else return {count, 0} -- CLEAN end
Memory at scale
At 1M clicks/sec, the fraud state store could grow fast. With a 60s window and 120s TTL, we maintain ~60M active keys consuming ~15GB across shards. HyperLogLog for Rule 2 adds only ~12KB × (unique IPs in window) — roughly 2-3GB. Total fraud state memory: ~18GB, well within a 10-node Redis cluster.
Cold path — daily reporting

Generates daily reports for advertisers showing clicks, impressions, spend, and fraud deductions. Runs as a batch job at 02:00 UTC with a 30-minute SLA.

Pipeline stages

1. Micro-batch writer: Consumes from the same Kafka topic with a separate consumer group. Buffers clicks into 5-second micro-batches and writes Parquet files to GCS/S3. Each file contains ~5M events, partitioned by date/hour/advertiser_bucket.

2. Schema: Each Parquet row: click_id, advertiser_id, campaign_id, ad_id, timestamp, cost_micros, ip, user_agent, is_fraudulent, geo_country. Columnar format compresses ~100 bytes/event → ~12 bytes/event. Daily: 86B × 12 bytes ≈ 1TB compressed.

3. Daily rollup: A Spark/BigQuery job at 02:00 UTC runs GROUP BY advertiser_id, campaign_id, date aggregating clicks, impressions, gross spend, fraud clicks, and net spend. Joins with the fraud flags table to exclude fraudulent clicks from billing. 86B rows → ~2M advertiser-campaign-day rows in ~30 minutes.

4. Report delivery: Rolled-up data lands in a serving table (Spanner/CloudSQL) that powers the advertiser dashboard API. Advertisers can query by date range, campaign, and ad group.

daily_rollup.sql
INSERT INTO advertiser_daily_report SELECT advertiser_id, campaign_id, DATE(timestamp) AS report_date, COUNT(*) AS total_clicks, COUNT(CASE WHEN NOT is_fraudulent THEN 1 END) AS valid_clicks, SUM(CASE WHEN NOT is_fraudulent THEN cost_micros ELSE 0 END) / 1000000.0 AS net_spend, SUM(cost_micros) / 1000000.0 AS gross_spend, COUNT(CASE WHEN is_fraudulent THEN 1 END) AS fraud_clicks, COUNT(DISTINCT ip) AS unique_ips FROM click_events WHERE DATE(timestamp) = CURRENT_DATE() - 1 GROUP BY advertiser_id, campaign_id, report_date;
Why Parquet over raw JSON/CSV?
Columnar format gives ~8-10× compression over row-oriented JSON. BigQuery reads only the columns referenced in each query — a rollup query touching 5 columns out of 10 reads ~50% less data, saving both time and cost. Predicate pushdown on date/advertiser partitions further reduces scan volume by 95%+ for typical queries.
Failure modes & mitigations

For every component, we define the failure behavior, the default strategy (fail-open vs fail-closed), and the recovery mechanism.

Scenario Strategy Behavior Recovery
Redis shard down Fail-open Continue serving ads, log clicks to Kafka. Brief overspend during outage. On recovery, replay un-decremented clicks from Kafka log using last committed offset.
Budget enforcer crash Fail-open Kafka consumer group rebalances partitions to surviving instances within ~30s. New consumer queries Redis for all advertisers at/below threshold to pre-populate blocklist. Cold start ~200ms.
Kafka broker down Fail-open Replication factor 3 ensures no data loss. Consumers failover to replica leaders. ISR (in-sync replicas) elect new leader. Producers retry with idempotent delivery (exactly-once semantics).
Fraud detector lag Fail-open Clicks still process on hot path. Fraud flags arrive late — refunds issued retroactively. Consumer lag monitored via Burrow. If lag > 5 minutes, scale up fraud detector instances.
Cold path writer fails Fail-open Kafka retains events for 7 days. No data loss. Daily report delayed. Restart writer from last committed offset. Backfill Parquet files. Report SLA breach triggers PagerDuty.
Double-decrement bug Prevented Lua script checks SISMEMBER(click_id) before decrementing. Duplicate clicks are no-ops. 5-minute TTL on dedup set bounds memory. Clicks older than 5 min are guaranteed committed.
Why fail-open is the right default
The alternative — fail-closed (stop serving all ads on any component failure) — costs advertisers money in lost impressions and costs Google revenue. A brief period of overspend during an outage is economically recoverable (refund the overspend). Lost impressions during a false-positive shutdown are not. The exception is fraud detection, where we err on the side of letting suspicious clicks through rather than blocking legitimate ones — false positives damage advertiser trust.

Reconciliation loop

Why it exists: Redis is fast but not durable. If a shard dies, we lose in-memory budget counters. The reconciler ensures we always have a recent checkpoint to recover from.

How it works: Every hour, a reconciler service reads all budget counters from Redis and writes them to a durable database (Spanner). On Redis recovery, we reload from the last Spanner checkpoint, then replay Kafka events from that checkpoint forward to rebuild exact state.

The gap: We lose at most 1 hour of decrements on a full Redis failure. For a worst-case advertiser spending $10K/day, that's ~$417 of untracked spend. Acceptable given the rarity of full shard failures with replicas.

EM angle

How you'd staff this, roll it out, handle incidents, and run it in production — the signals that separate an EM-level answer from a pure IC design.

Staffing
Team structure (6-8 engineers)
2 backend engineers on the hot path (budget enforcer + Redis). 1 backend on the warm path (fraud detection). 1 data engineer on the cold path (Parquet pipeline + BigQuery). 1 infra/SRE engineer on Kafka + Redis operations. 1 frontend engineer for the advertiser reporting dashboard. Critical path: hot path must ship first (4 weeks), warm path second (2 weeks), cold path last (3 weeks). Hot and cold paths can parallelize — warm path depends on hot path's Kafka topic.
Rollout
Phased deployment strategy
Phase 1 — Shadow mode: Run budget enforcer alongside existing system, compare results, don't enforce. 1 week.
Phase 2 — Canary (1%): Enforce on 1% of traffic (small advertisers first). Monitor overspend, latency, false positives. 1 week.
Phase 3 — Ramp: 1% → 10% → 50% → 100% over 2 weeks, with automated rollback on error rate > 0.1% or p99 latency > 15ms.
Incident handling
On-call + runbooks
Runbook 1 — Budget overspend alert: Check Redis shard health → verify STOP propagation latency → check for partition rebalancing.
Runbook 2 — Fraud detection lag: Check consumer lag via Burrow → scale up instances → verify Redis sorted set memory usage.
Runbook 3 — Report SLA breach: Check Spark job logs → verify Parquet file integrity → manual backfill from Kafka if needed.
On-call rotation: 2 engineers, 1-week shifts, 15-minute response SLA for P0.
Monitoring
SLO dashboard
SLO 1: Budget enforcement latency p99 < 10ms (measured: Kafka consume to Redis response).
SLO 2: Budget overspend per advertiser < 0.5% of daily budget.
SLO 3: Fraud detection window < 5s (measured: click timestamp to flag emission).
SLO 4: Daily report available by 03:00 UTC (1-hour SLA after 02:00 UTC job start).
Dashboards: Grafana for real-time metrics, BigQuery for historical SLO compliance reporting.
Cross-team dependencies
This system crosses the Ads Serving, Ads Billing, and Trust & Safety teams. The hot path gate integrates directly with Ads Serving's ad selection pipeline — I'd set up a shared design doc and weekly sync with their tech lead. The fraud detection output feeds into Trust & Safety's existing abuse pipeline — we need to align on signal format and SLAs. Billing team owns the advertiser dashboard — we deliver the rollup table, they own the API and UI. I'd establish clear API contracts (protobuf schemas) at each boundary and a shared Slack channel for incident coordination.

Risk register

Technical risk — Redis single point of failure: Mitigated by Redis Sentinel + replicas per shard. If a master fails, Sentinel promotes a replica in ~5 seconds. Runbook: if Sentinel fails to promote, manually failover and notify on-call. Probability: low. Impact: high (ads stop or overspend).

Product risk — false positive fraud detection: Blocking legitimate clicks damages advertiser trust. Mitigated by setting conservative thresholds (5 clicks/60s is generous) and making Rule 3 (CTR anomaly) manual-review only, never auto-block. We track false positive rate weekly and adjust thresholds quarterly.

People risk — knowledge concentration: If the 2 hot-path engineers both leave, we lose critical Redis Lua expertise. Mitigated by requiring runbooks for every operational procedure, pair programming on the Lua scripts, and quarterly rotation of on-call responsibilities so everyone touches each component.
