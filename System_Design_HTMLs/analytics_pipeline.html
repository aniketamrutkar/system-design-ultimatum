Layer 01

Event Collection &
The Clickstream

Every meaningful user action produces an event. The challenge is not collecting events — that's easy. The challenge is doing it without adding latency to the user-facing critical path, and without losing events when services restart.

Client
Player SDK + UI event layer
web · iOS · Android · TV — local IndexedDB/SQLite queue, batched every 10s or 20 events
batch POST over HTTPS · retry on failure
Collector
Event collector service
schema validation · timestamp correction · IP geolocation enrichment · Redis bloom filter dedup
Kafka producer · acks=1 for high volume · acks=all for critical events
Bus
Kafka analytics topics
view.event · ui.event · perf.event · error.event · experiment.exposure
three independent consumer paths — same events, different latency SLAs
Sinks
Flink · BigQuery · Real-time dashboards
seconds latency · 60s latency · Parquet/GCS batch → warehouse

The most important engineering decision here is the local event queue on the client. Events are not sent immediately — they're written to IndexedDB on web or SQLite on mobile, then flushed in batches. This means: if the user loses connectivity for 30 seconds while watching, events accumulate locally and are sent when connectivity returns. The analytics pipeline never loses events to network blips. The critical playback path is never blocked waiting for an analytics write.

Layer 02

Apache Flink —
Real-time Stream Processing

Flink reads from Kafka and produces aggregated results in seconds to minutes. It solves three hard problems that naive Kafka consumers get wrong every time: windowing, late data, and durable state.

Window types

Tumbling
Fixed non-overlapping buckets
Each event belongs to exactly one window. Used for: view counts per minute, hourly active users, daily upload totals. Output fires once at window close. Predictable, simple, lowest overhead.
Sliding
Overlapping rolling windows
60s wide, slides every 10s. Each event belongs to multiple windows. Used for: CDN stall rate monitoring, trending score. More expensive but gives fresher signal for automated failover alerting.
Session
Dynamic gap-based windows
Closes after 30 min of inactivity. Groups all events from one browsing session. Dynamic window size based on engagement. Used for: session length, funnel completion, entry surface attribution.
Global + Trigger
Threshold-based unbounded window
FIRE
Accumulates views since publish. Fires when velocity crosses 10K / 100K / 1M threshold. Used for trending detection. Window stays open as long as the video exists — not time-bounded.

Watermarks — handling late events. Events don't arrive in order. A view from a user on a plane arrives hours late. We use forBoundedOutOfOrderness(Duration.ofMinutes(2)) with server_ts (not client_ts — client clocks can be wrong by hours). A window covering [10:00, 10:01] closes when the watermark reaches 10:03:00. Events arriving after that go to a view.event.late side output — a slower correction job issues BigQuery updates. Real-time dashboards undercount slightly; warehouse is eventually correct.

Production Flink jobs

view-counter-realtime
tumbling 60s · keyed by video_id · 32 task managers
Reads view.event. Writes INCRBY video:views:60s:{id} to Redis. Drives the live view counter visible on the video page. State: 800M keys × 8 bytes = 6.4 GB across task managers.
~5s latency
cdn-health-monitor
sliding 60s/10s · keyed by cdn_pop × provider
Reads perf.event. Computes avg stall_ms, P95 startup_ms per PoP. If avg stall >500ms: publishes cdn.degraded → CDN orchestrator shifts traffic weight automatically. Automated CDN failover driven by real QoE.
<15s action
trending-detector
global window + trigger · keyed by video_id
Accumulates views since publish. Fires when velocity_score = views_last_60m / max(1, views_prev_60m) exceeds 10×. Publishes video.trending → recommendation boost + creator notification.
5 min refresh
experiment-metrics-realtime
tumbling 5-min · keyed by experiment × variant
Joins view.event + experiment.exposure on user_id. Computes avg watch_pct, CTR, stall_ms per variant. Writes to experiment dashboard. Gives owners live signal within 10 min of launch — early harm detection.
<10 min
feature-store-updater
session window · keyed by user_id · 64 task managers
Maintains rolling feature vector per user: avg_watch_pct_7d, topic_affinity_vector, creator_affinity_map, recent_searches. Writes to Redis Feature Store every 5 min. Feeds recommendation deep ranker. State: ~1 TB with RocksDB.
<6 min
Layer 03

BigQuery —
The Data Warehouse

Where everything lands for deep analysis — the kind that takes seconds to minutes because it's scanning billions of rows across months of history. Columnar storage means you only read the columns you query.

RT
Real-time path
Flink sink writes micro-batches every 30s to BigQuery streaming inserts. Events queryable within ~60s of arriving. Used for experiment.exposure — experiment analysts need data within minutes of launching a test. Higher cost (per-row billing).
BT
Batch path
Kafka → GCS (Parquet files every 5 min via Kafka Connect) → BigQuery external table → nightly INSERT INTO job. Used for high-volume view.event. Cheaper — no per-row streaming billing. Events queryable with up to 6h lag.
P+C
Partitioning + clustering
Partitioned by DATE(server_ts). Clustered by (country, device_type). Without: query over one day scans 11B rows. With: same query scans ~200M rows. 50× cost and speed improvement — the single most impactful schema decision.

Query governance. At 11B events/day, storage is manageable (~$50K/month) but ad-hoc queries without proper partition filters can cost thousands of dollars per query. The solution: require all production queries to include a WHERE DATE(server_ts) BETWEEN clause, enforced through a query linting tool in the SQL IDE analysts use. One engineer forgetting a filter can cost $5,000 in a single query.

Layer 04

A/B Testing —
The Decision Engine

Where the analytics pipeline pays for itself. Every major product decision is validated through controlled experiment before full rollout. Does the new recommendation algorithm increase watch time? Does the dark theme reduce session churn?

01
Assignment
When a user loads the app, experiment service hashes user_id + experiment_id. Hash maps to bucket 0–999. Bucket <500 → control. Bucket ≥500 → treatment. Same user always gets same bucket — deterministic, no coordination needed across servers. Every server independently computes the same assignment.
02
Exposure logging
experiment.exposure event fires when user actually sees the experimental surface — not when assigned. This is the intent-to-treat vs as-treated distinction. A user assigned to "new search ranking" who never searches during the experiment period is excluded from analysis. Dilution avoided.
03
Analysis
Join experiment.exposure with view_events on user_id. Group by variant. Compute metric (avg watch_pct, P95 stall_ms, CTR). Run two-sample t-test or Mann-Whitney U test. Ship if p <0.05 AND effect is practically meaningful — statistical significance alone is not sufficient.

Three pitfalls that break naive A/B testing

Novelty effect
Users behave differently when they first see new UI. First 3–7 days show inflated positive metrics — users engage because it's new, not because it's better. The dashboard flags early results as provisional and shows a separate "steady state" metric excluding the novelty period.
Network effects
If the experiment changes recommendations and user A shares a video with user B, the experiment has spread beyond the treatment group — violating the independence assumption. For experiments with potential network effects, use geo-based randomization: entire cities assigned to control or treatment.
Multiple comparisons
Run 100 simultaneous experiments with p<0.05 → expect 5 false positives by chance alone. The platform applies Bonferroni correction (divide threshold by number of concurrent experiments) and enforces a maximum of 30 concurrent experiments per surface.
Reliability

Failure Modes

Failure Sev Immediate effect Recovery
Event collector down P1 Client events queue locally (IndexedDB/SQLite). No data lost — flush when collector recovers. Analytics lag grows. Stateless service — restart instances. Clients retry automatically. Usually <60s to full recovery.
Flink task manager crash P1 Partitions reassigned to surviving managers. Checkpoint recovery restores state from last GCS snapshot. Kafka replay covers the gap. 30–90s recovery. Exactly-once semantics — no double-counting. Consumer lag spike resolved as TM catches up.
cdn-health-monitor job down P0 Automated CDN failover stops. A degraded PoP continues serving traffic — users in that region experience elevated stall. No automatic recovery. Manual CDN weight shift while Flink job restarts. P0 because impact is user-facing and invisible without the automated signal.
BigQuery partition scan runaway P2 Analyst query without partition filter scans full history. Single query can cost $2,000–5,000 and slow down the warehouse. Query cancelled after cost threshold. Linting rule enforced at submission — prevents the query from running at all.
Kafka consumer lag spike on view.event P1 View counts stale. Trending detector fires late. Real-time dashboard shows old data. Scale up Flink task managers (up to partition count limit). Fix slow downstream. Alert at 100K message lag threshold.
Experiment analysis query returns wrong results P1 Team ships experiment based on incorrect data. Feature deployed broadly — could harm user experience at scale. All experiment queries logged. Replay from Kafka and recompute. Offset reset makes reprocessing straightforward. Peer review required before ship decision.
EM Perspective

Staffing, Cost &
Operations

Team structure (6–8 engineers)
Platform team (3–4): Flink jobs, Kafka infrastructure, BigQuery schema, data reliability. Product analytics (2–3): experiment design, analysis tooling, dashboards, metric frameworks. Data science (1–2): ML feature engineering, model evaluation infrastructure. The hardest hire: stream processing engineers who also understand statistics and experiment design — these three disciplines rarely overlap.
Cost profile
BigQuery storage at 11B events/day: ~$50K/month manageable. Flink cluster for 5 production jobs: ~$30K/month (GCP c2 instances, RocksDB on local SSD). Kafka cluster: shared with pipeline, ~$15K/month. Total analytics infrastructure: ~$100K/month. Main cost lever: Flink over/under-provisioning. Autoscale based on consumer lag — idle compute is the biggest waste.
Flink cluster sizing
The most frequent source of analytics incidents. Under-provisioned task managers run out of RocksDB state space — job falls behind, cdn-health-monitor stops firing, CDN problems go undetected. Over-provisioned clusters waste $100K/year in idle compute. Right approach: continuous autoscaling based on Kafka consumer lag. If lag on any input topic >100K messages: add task managers. If lag is consistently zero: scale down.
On-call responsibilities
Analytics on-call has two classes of alert. Operational (Flink job down, consumer lag, BigQuery write failure) — wake-up-at-3am severity. Data quality (experiment metrics diverged, view counts anomalous, session attribution broken) — next-business-day severity. The key operational discipline: never let a data quality issue ship to production without a peer review. One wrong experiment decision can mean rolling back a feature that 500M users already have.
