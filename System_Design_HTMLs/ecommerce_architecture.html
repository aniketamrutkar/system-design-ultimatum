E-Commerce System Design

Amazon / Flipkart scale · 500M DAU · Global · Full marketplace + logistics + payments

System Overview

We are designing a global-scale e-commerce super-app at the level of Amazon or Flipkart — supporting 500M daily active users across buyers, sellers, admins, affiliates and 3rd party API partners, covering all product verticals with built-in logistics and payments.

The core engineering challenges are: search at 290K queries/sec, inventory consistency under flash sales, exactly-once payment processing, and distributed order sagas across 14 independent microservices.

Core design decisions

Microservices with saga pattern

14 independently deployable services. Distributed transactions via choreography-based saga over Kafka — no 2-phase commit. Compensating transactions on failure.

Hybrid push-pull for inventory

Real-time stock reservation via Redis pessimistic locks during checkout. Eventual sync to PostgreSQL. Prevents oversell without blocking catalog reads.

Elasticsearch + ML re-ranking

BM25 full-text search + vector similarity for semantic results + ML re-ranking for personalization. 32-shard cluster, updated via Kafka consumers.

Multi-region active-active

4 regions: US-East, EU-West, India, SEA. Geo-routing at DNS level. Data residency compliance per region (GDPR, DPDP, CCPA).

Key clarifying questions

Inventory model

Real-time reservation (not eventual) — overselling is catastrophic for seller trust and customer experience. Pessimistic locking during checkout window.

Fulfillment model

Hybrid: FBA-style (our warehouse) + self-ship by seller. Logistics service abstracts both behind a unified carrier assignment API.

Flash sale approach

Queue-based virtual waiting room, not optimistic locking. Prevents thundering herd on DB while maintaining fairness.

Physical goods only

No digital goods for now. Simplifies fulfillment, inventory, and tax calculation. Digital goods would require license key management and DRM.

Flash Sale Engine

The hardest single problem in e-commerce infrastructure. Big Billion Day drives 50x normal traffic in under 60 seconds of a sale going live.

Problem: Thundering herd

50M users simultaneously hitting a product page at T=0. Naive implementation DDOSes the DB, causes oversell, and creates an unfair experience where only users with fast connections win.

Solution: Virtual waiting room

Traffic enters a queue before reaching checkout. Position in queue determined by arrival time. Users see real-time position. Checkout tokens issued at a controlled rate matching inventory depletion speed.

Architecture

User hits flash sale URL at T=0
→
Gateway routes to queue service
→
Redis LPUSH into sale queue
→
User polls position via SSE
→
Token issued when position reached
→
Checkout with token (5 min TTL)
→
Lua script atomic stock decrement
Pessimistic locking (Redis SETNX) vs optimistic (compare-and-swap) vs queue: at 50x traffic, optimistic locking causes extreme retry storms. Pessimistic locking serializes too aggressively. Queue-based approach provides fairness + controlled throughput + survives traffic spike without DB pressure.
Redis primary failure during flash sale: Redis Sentinel promotes replica in <30s. Queue state is preserved (AOF persistence). Users experience a brief wait but no data loss. Fallback: serve static "sale ended" page from CDN if queue service is fully down.
Flash sale infra is pre-warmed 2 hours before go-live. Load test at 2x expected peak the day before. Dedicated on-call rotation during Big Billion Day. Auto-rollback trigger if error rate >1% within 60s of sale start.

Non-Functional Deep Dive

Security

PCI-DSS Level 1

Payment card data never touches our servers. Tokenized via Stripe / Razorpay. All cardholder data flows directly to PCI-compliant gateway. We store only the last 4 digits + token.

Auth architecture

OAuth 2.0 + PKCE for social login. JWT access tokens (15 min). Refresh tokens in HttpOnly cookies (30 days). mTLS between all internal services. API keys for seller/3P clients.

Encryption

TLS 1.3 for all external traffic. AES-256 at rest for sensitive fields. KMS-managed keys per region. PII fields (email, phone) encrypted at column level in PostgreSQL.

Secrets management

HashiCorp Vault for all secrets. No env vars in code. Automatic rotation for DB credentials every 24h. Audit log for every secret access.

Multi-region & Cross-DC

Active-active across 4 regions. Geo DNS routes users to nearest region. Each region is fully self-sufficient for reads. Writes for orders/payments use leader-follower with US-East as primary — asynchronous replication to other regions (RPO ~1s for non-financial data, RPO=0 for payments via synchronous cross-region write).

Cross-DC failover: If a region goes down, Route53 health checks detect within 30s and reroute traffic. Automated runbook via AWS FIS (Fault Injection Service) tested monthly in GameDay exercises.

Key Tradeoffs

Microservices vs Monolith

Chose microservices for independent scaling, independent deployment, and team autonomy at 1000+ engineer scale. Cost: operational complexity, distributed tracing needed, network latency between services. Mitigated by service mesh (Istio) and strong observability.
Eventual consistency for catalog

Product catalog (prices, descriptions) uses eventual consistency. A buyer might see a price that's 2s stale. Acceptable — we show a "price confirmed at checkout" message. Allows 100x read throughput via CDN caching without DB reads.
Kafka choreography vs orchestration

Saga choreography chosen for loose coupling between order, payment, inventory services. Each service reacts to events independently. Downside: harder to visualize overall flow. Mitigated by distributed tracing and a separate saga state monitor service.
Redis inventory locks vs DB transactions

Redis pessimistic locks for inventory reservation (not DB SELECT FOR UPDATE). Reason: Redis handles 500K ops/sec vs PostgreSQL's ~50K. Lock TTL = 10 min (checkout window). Risk: lock orphaning on crash — mitigated by TTL expiry and background reconciliation job.
MongoDB for catalog vs PostgreSQL

Different product categories have wildly different attributes (a book has ISBN, a shirt has size/color, a laptop has RAM/CPU). JSONB in PostgreSQL is an alternative but MongoDB's native document model and horizontal sharding story are stronger at 500M SKUs.
Build vs buy: search

Elasticsearch chosen over Algolia (too expensive at this scale, ~$500K/month), AWS OpenSearch (same engine, better control with self-managed), and Solr (worse operational tooling). At 500M SKUs and 290K QPS, Elasticsearch + self-managed gives best cost-performance.

Key risks & mitigation

Payment gateway outage
Mitigation: 3 gateway providers in active-active (Stripe primary, Razorpay fallback, UPI direct fallback). Circuit breaker auto-switches in <5s. Tested monthly.
Inventory data inconsistency
Mitigation: Nightly reconciliation job comparing Redis locks vs PostgreSQL actual stock. Alert on delta >1%. Manual review process for disputed stock counts.
Flash sale DDoS (self-inflicted)
Mitigation: Virtual waiting room absorbs spike. Pre-scaled K8s nodes 2h before sale. CDN serves product page; only checkout hits origin. Rate limit by user_id at gateway.
Data breach / PII leak
Mitigation: Column-level encryption for PII. No PII in logs (scrubbed at ingestion). Vault-managed keys. Annual penetration test. Bug bounty program.
