Core design decisions

Microservices + Saga pattern

14 independently deployable services. Distributed transactions via choreography-based saga over Kafka — no 2-phase commit. Compensating transactions on failure. Outbox pattern for guaranteed event delivery.

Hybrid consistency model

Strong ACID for orders and payments (RPO=0). Eventual for catalog, search, recommendations. Redis pessimistic locks for inventory reservation. CAP theorem applied per domain, not globally.

Elasticsearch + ML re-ranking

BM25 full-text + dense vector kNN for semantic search + LightGBM re-ranking for personalization. 32-shard cluster, real-time index updates via Kafka consumers. p99 <200ms.

Multi-region active-active

4 regions: US-East, EU-West, India, SEA. Geo-DNS routing. Data residency compliance per region (GDPR, DPDP, CCPA). RPO=0 for financial data via synchronous cross-region replication.

Key clarifying questions

Inventory model?

Real-time reservation — not eventual. Overselling destroys seller trust and creates legal refund obligations. Pessimistic Redis locking during checkout window (10 min TTL).

Fulfillment model?

Hybrid: FBA-style (our warehouse, we pick-pack-ship) + self-ship by seller. Logistics service abstracts both behind unified carrier assignment API. Seller chooses per product.

Flash sale approach?

Queue-based virtual waiting room — not optimistic locking. Prevents thundering herd on DB while maintaining first-come-first-served fairness. Checkout token with 5-min TTL.

Physical goods only?

Yes for Phase 1-2. No digital goods. Simplifies fulfillment, inventory model, and tax calculation. Digital goods deferred to Phase 3 (requires license key mgmt and DRM).

Catalog indexing pipeline

Product update → Kafka catalog.updated → ES indexer consumer → bulk API (500 docs/batch, 1s interval) → index lag <2s. Separate trending index updated every 30s for autocomplete and trending search. Full re-index possible via snapshot restore without downtime (blue-green index aliases).

TRADEOFF ES vs Algolia vs separate vector DB: Algolia is excellent but ~$500K/month at this scale. A separate vector DB (Pinecone, Weaviate) adds operational complexity with two search systems. ES 8.x dense_vector + kNN handles our semantic search needs within one system. Separate vector DB becomes compelling only above 100K vector updates/sec — we're at ~5K/sec.

Flash Sale Engine

The hardest single problem in e-commerce infrastructure. Big Billion Day drives 50x normal traffic in under 60 seconds. Click each step for implementation details.

THE PROBLEMThundering herd at T=0: 50M users simultaneously hitting a product page. Naive implementation DDOSes the DB, causes oversell, and rewards users with fast internet connections over genuine first-comers. Both a technical failure and a fairness failure.
THE SOLUTIONVirtual waiting room: Traffic enters a queue before reaching checkout. Position = arrival time. Users see real-time position via SSE. Checkout tokens issued at controlled rate matching inventory depletion. Fairness guaranteed — 3G user and fiber user at same position get same chance.

Systems Thinking

What separates Solid from Outstanding on the Google rubric. The system doesn't exist in isolation — it exists within an organization, a business, and a technology ecosystem that is constantly evolving.

Organizational impact

Conway's Law in action

The architecture you design becomes the org chart. 14 services = 14 team ownership boundaries. Each service needs an owner — on-call accountability, feature roadmap, SLO ownership. This is not a coincidence; it's the design intent.

Implication: the platform team (API gateway, Kafka, K8s, observability) must be staffed before product teams arrive. If product engineers land first, they rebuild shared infrastructure independently — creating the distributed monolith anti-pattern.

Developer velocity impact

The first 6 months are slower than a monolith. Engineers deal with network failures that don't exist locally, Docker Compose multi-service setup, distributed traces instead of local stack traces, and service contract negotiation.

Velocity recovers at month 8-12 as tooling matures. By month 18, teams ship independently without deploy coordination. This is the payoff — 50 teams deploying daily without blocking each other.

Technical debt to manage proactively

DEBT RISK 1API contract drift: Every service that ships without an OpenAPI spec + consumer-driven contract tests (Pact) creates hidden coupling. At 14 services today, 40+ in 3 years — unspecified contracts become the primary source of production incidents. Contract-first must be a Day 1 engineering standard.
DEBT RISK 2MongoDB schema drift: Without a catalog schema registry, 500M documents accumulate inconsistent field names for the same concept across different seller integrations. Schema registry + validation on ingestion is a 6-month investment that saves years of data quality pain.
DEBT RISK 3Analytics instrumentation: Every Phase 2/3 ML feature depends on structured event data from Phase 1. Missing fields or inconsistent event schemas in Phase 1 means 2-3 months of data cleaning before ML work can begin. Analytics schema is a Phase 1 deliverable.
DEBT RISK 4On-call runbook completeness: A service whose only runbook is "call the person who built it" is a single point of organizational failure. Every service must have a runbook complete enough for a new engineer to handle a P1 in their first week.

Adjacent systems this enables

Fintech (BNPL, digital wallet)

Payment service + user KYC data + order history = credit scoring foundation. A "Buy Now Pay Later" product is 3 months of engineering on top of existing infrastructure. Not a new platform — an extension.

Advertising platform

Seller-sponsored listings require an ad auction service. The search re-ranking layer already has a slot for ad injection — the infrastructure exists. The business model and auction logic are the engineering work, not the plumbing.

B2B marketplace

Large enterprise buyers need bulk ordering, purchase orders, credit terms. The order service can be extended; the seller service needs a B2B tier. 6-month project in Phase 3 — same data model, different business rules.

Quick commerce (grocery)

Same platform, different SLAs. Grocery needs 30-minute delivery, real-time store-level inventory, dark stores, bike couriers. The logistics service abstraction handles this — a new carrier type, not a new system.

Where this system fits in the broader ecosystem

The e-commerce platform is not just a product — it is the data flywheel for the entire company. Every order, search, click, and return generates behavioral data that feeds: seller analytics (ClickHouse), financial reporting (Snowflake), customer behavior analysis (data lake on S3), and ML models (recommendation, fraud, pricing).

The Kafka event bus is the integration point for every future business unit. A new analytics use case, a new ML model, a new business line — all consume from Kafka without the e-commerce platform needing to change. The platform is a data producer. Every future product is a consumer. This is the most important architectural decision in the entire system — and the most invisible one.

Non-Functional Requirements Deep Dive

Security, privacy, multi-region architecture, and capacity planning in implementation depth — not just checklist items.

Security architecture

Auth layers

Layer 1 — Edge (Who are you?): JWT access tokens (buyers/sellers/admins, RS256, 15min TTL), API keys (3rd party, scoped per endpoint), mTLS certificates (service-to-service, rotated every 24h via cert-manager + Vault).

Layer 2 — Authorization (What can you do?): RBAC enforced at service level, not just gateway. Buyers: read:products, write:cart/orders. Sellers: write:own-catalog, read:own-orders. Each service validates JWT claims independently — no central auth SPOF.

Layer 3 — Resource ownership: Order service validates order.user_id == jwt.user_id. Valid JWT is not enough — you can only access your own data.

PCI-DSS Level 1

Tokenization flow: Browser → Stripe.js iframe (card never touches our DOM) → Stripe returns payment_method_token → checkout sends token → payment service calls Stripe API. We store: gateway_token, last4, brand, expiry only. Raw card data never in our systems.

CDE isolation: Payment service in dedicated VPC with no internet egress. No other service calls into CDE directly. Network segmentation enforced at security group + NACLs level.

Annual QSA audit + quarterly pen test for payment flows. Bug bounty on HackerOne ($500–$50K per finding).

Encryption

TLS 1.3 for all external traffic (1.2 explicitly disabled). mTLS for all internal service-to-service. AES-256 at rest for PII fields (column-level encryption in PG). KMS-managed keys, separate key per data tier. PII fields encrypted: email, phone, full name, address, national ID.

Secrets management

HashiCorp Vault as central secrets store. DB credentials auto-rotated every 24h — services fetch dynamically at startup. No hardcoded passwords in env vars, Docker images, or Git. Pre-commit hooks + GitGuardian scanning. All secret access logged and auditable.

Tradeoffs Summary

Every major architectural decision — what we chose, what we rejected, exactly why, and what conditions would make us change our mind. This is what Outstanding looks like on the Google rubric.

Architecture style

TRADEOFF 1 — Microservices vs Monolith

Chosen: Microservices (14 services, not 140 — domain-aligned, not function-aligned).

Rejected: Monolith (right for <20 engineers, wrong for 1000+). Modular monolith (right for 20-100 engineers — valid intermediate choice).

What we gave up: Developer experience in early phases. New engineers deal with Docker Compose complexity, distributed tracing, and service contracts before writing their first feature. 6 months slower than a monolith.

What would change our mind: If team permanently stays under 50 engineers, a modular monolith delivers 80% of the benefits at 20% of the operational complexity.

Consistency model

TRADEOFF 2 — Strong vs Eventual consistency (per domain)

We did not make a blanket choice. CAP theorem applied per domain:

DomainModelReasoning
Orders, paymentsStrong (ACID)Financial record. A missed order or double charge is a legal and trust failure.
Inventory reservationStrong (Redis atomic)Oversell destroys seller trust and creates refund obligations.
Inventory displayEventual (~2s)"500 left in stock" being 2s stale is acceptable UX. Allows 100x read throughput via caching.
Product catalogEventual (~2s)Price being 2s stale acceptable — "price confirmed at checkout" message shown.
RecommendationsEventual (~1h)Personalization model updated hourly. "Good enough" not "perfect" — acceptable UX.

Distributed transactions

TRADEOFF 3 — 2-Phase Commit vs Saga

Rejected — 2PC: Coordinator failure leaves all services locked indefinitely. Network partitions (constant at this scale) cause indefinite blocking. Correct for 2-service systems, catastrophic for 14+.

Chosen — Saga choreography: Each service reacts to Kafka events independently. On failure, compensating transactions undo completed steps. No central coordinator = no single point of failure.

What we gave up: Harder to debug. Transaction state is distributed across services and topics. Mitigated by a dedicated saga state monitor service that reconstructs state from event history.

What would change our mind: If debugging becomes a major pain point at scale, evaluate Temporal for orchestrated sagas — visual workflow, explicit state machine, retry policies built-in. The tradeoff is coupling to the Temporal cluster.

Inventory locking

TRADEOFF 5 — Redis locks vs DB transactions

Why not PostgreSQL SELECT FOR UPDATE: PG handles ~50K TPS. Redis handles 500K+ ops/sec. At 1,150 reservations/sec avg and 11,500/sec during flash sale — Redis is the right tool. PG would be the bottleneck at 10x traffic.

Risk accepted: Lock orphaning on Redis crash (mitigated by TTL expiry — locks release within 10min maximum). Slight inconsistency window between Redis state and PG source of truth (mitigated by nightly reconciliation job, alert on delta >1%).

What would change our mind: If inventory accuracy SLA tightens to zero tolerance, evaluate CockroachDB or Google Spanner for globally consistent inventory — at significantly higher cost and latency.

  • Cart (add, remove, update quantity)
  • Checkout — single address, Stripe card payment only
  • Order management — place, view, cancel
  • Email notifications (SES — order confirmation, shipping)
  • Seller onboarding — manual KYC, basic listing form
  • Admin panel — order management, basic moderation

Key risks and mitigation

RISK — Payment gateway outageProbability: Medium · Impact: Critical

Mitigation: 3 providers in active-active (Stripe primary, Razorpay fallback, UPI direct fallback). Circuit breaker auto-switches in <5s. Tested monthly. No single provider handles >60% of volume.
RISK — Flash sale DDoS (self-inflicted)Probability: High · Impact: High

Mitigation: Virtual waiting room absorbs spike. Pre-scaled K8s 2h before. CDN serves product page (only checkout hits origin). Rate limit by user_id at gateway. Auto-rollback if error rate >1% for 60s.
RISK — Inventory data inconsistencyProbability: Medium · Impact: High

Mitigation: Nightly reconciliation compares Redis locks vs PG actual stock. Alert on delta >1%. Background job releases expired locks every minute. Manual review process for disputed counts.
RISK — Key engineer departureProbability: High · Impact: Medium

Mitigation: Service ownership at team level, not individual level. Runbook complete enough for new engineer to handle P1 in first week. No single-person-owned services enforced via team policy.
