Google Photos — System Design

PROBLEM STATEMENTDesign a global-scale photo storage and management platform serving 1B+ MAU / 500M DAU. Engineering challenges: (1) ingest 4B photos/day at 162 GB/s sustained, (2) serve trillions of variants with >97% CDN hit rate at p99 <200ms, (3) run ML pipelines (CLIP, FaceNet, YOLO) without touching the write-path latency, (4) zero data loss on irreplaceable personal memories — 11 nines durability.

Five Core Architectural Decisions
  • Content-addressed storage — photo_id = SHA-256(bytes) → dedup + immutable URLs + infinite CDN TTL, no invalidation
  • 4-tier cache hierarchy — client LRU → CDN edge → regional NVMe/Redis → origin Colossus
  • Async ML pipeline — CLIP/FaceNet/YOLO runs off write-path via Kafka; photos visible in grid immediately, searchable in ~30s
  • Saga orchestration — upload is a 3-phase saga (initiate→transfer→commit); compensating actions on each failure
  • Write-once semantics — photos never mutated; edits produce new derived objects; original preserved forever

System Architecture — 8 Layers

ARCHITECTURAL PATTERNWrite-fan-out + Read-cache-first. Every upload flows: resumable ingestion service → blob commit to Colossus → Kafka fan-out to async consumers (transcode, ML, index, cache-warm). Every read checks 4 cache tiers before touching origin. The metadata plane (Bigtable) and media plane (Colossus/GCS) scale independently. Kafka is the integration bus — write path decoupled from all downstream intelligence.

View Flow — 4-Tier Cache Traversal

KEY INSIGHTGrid open fires N parallel thumbnail fetches, one per visible photo. Each fetch independently traverses the cache hierarchy and short-circuits on first hit. The weighted average latency is ~52ms — well within the 200ms p99 SLO — because 90%+ of requests hit the first two client-side cache tiers with sub-10ms response.

Systems Thinking

Conway's Law — Architecture Determines Org Structure

Team Boundaries from Service Decomposition

The service split maps directly to team ownership. Upload Pipeline team owns TUS, dedup, safety scanner, ingest coordinator. Media Serving team owns CDN config, variant routing, cache warming, format negotiation. ML/Search team owns CLIP pipeline, ScaNN, FaceNet clustering, OCR. Storage Platform team owns Bigtable schema, Spanner, Colossus lifecycle policy, Redis cluster. Sharing & Social team owns album management, share links, contributor mode, notifications.

Non-obvious sequencing: Storage Platform must be hired and operational before Upload Pipeline. Bigtable row key design is the most irreversible schema decision in the entire system — making it wrong costs 18+ months to migrate at 4T rows. ML engineers must start in Phase 1 despite delivering no user-visible features — they need 6+ months of photo data before any model can be trained meaningfully.

Technical Debt to Manage Proactively

RISK — BIGTABLE ROW KEY LOCK-INThe row key (userID#reverseTimestamp#photoID) optimises for chronological grid reads. If primary access patterns shift to "all photos of person X" or "all photos at location Y", a re-key migration across 4T rows is a multi-year project. This is the single most irreversible architectural decision. Instrument access patterns from day 1; review at 500M photos.
RISK — ML INSTRUMENTATION DEBTEvery Phase 2 ML feature (Memories, smart albums, face confidence) depends on structured event data logged in Phase 1. Missing event fields = 3 months of data cleaning before model training begins. Analytics schema must be co-designed by ML engineers alongside Phase 1 product engineers. This is non-negotiable.
RISK — API CONTRACT DRIFT14+ microservices without OpenAPI specs + consumer-driven contract tests (Pact) = hidden coupling. At this service count, unspecified contracts are the #1 source of production incidents within 18 months. Enforce from day 1: CI gate rejects PRs that modify an API without a contract test update.
RISK — RUNBOOK GAPSThe ML pipeline and cache warmer are highest-risk for runbook gaps — most complex, least understood by non-ML engineers. A service whose runbook says "call the person who built it" is an org SPOF. Every service must have a runbook complete enough for a new engineer to handle a P1 in their first on-call week.

Non-Functional Requirements — Implementation Depth

Security Architecture — Three Layers

Layer 1 — Identity
  • JWT: RS256 (asymmetric — private key in Vault; public JWKS distributed to all services). Claims: sub (user_id), region, storage_tier, scopes[]. TTL: 1h.
  • Refresh tokens: HttpOnly + Secure + SameSite=Strict cookie. 30-day TTL. Rotated on every use. Stored in Spanner with revocation column. Revocation propagates in <5min via Redis pub/sub.
  • mTLS: All service-to-service calls. cert-manager + Vault. Certificate rotation every 24h. Upload Service → GCS uses GCP service account key, not mTLS.
  • Partner OAuth: Authorization code + PKCE. Scopes: photoslibrary.readonly / appendonly. Token introspection at API Gateway on every call. Rate limit: 10K req/day per app.
Layer 2 — Authorization
  • Resource ownership: photo.owner_user_id == jwt.sub enforced in Library Service (not just gateway) — gateway-only is insufficient for defense-in-depth.
  • Shared album access: Share token validated against Spanner shares table on every request. is_contributor checked before accepting write operations.
  • Partner scopes: Checked at API Gateway AND Library Service. API Gateway failure → second enforcement point catches it.
Layer 3 — Data Protection
  • TLS 1.3 minimum — 1.2 explicitly disabled via CDN config. Applied globally as of 2024 migration.
  • Encryption at rest: All Colossus objects encrypted with AES-256-GCM. Per-user key envelope managed by Cloud KMS. Face embeddings use a separate KMS key ring with stricter IAM.
  • PII columns: email, phone, name encrypted at column level in Spanner. Face embeddings never returned raw to any client — only cluster_id returned.
  • Secrets: Vault dynamic secrets, rotated every 24h. Zero env-var secrets enforced via CI secret-scanning (detect-secrets pre-commit hook).

Key Tradeoffs

TRADEOFF 1 — CONTENT-ADDRESSED STORAGE (SHA-256 AS PHOTO_ID) Chosen: photo_id = SHA-256(bytes). Enables: (1) instant dedup — same image from 2 devices = 1 blob, (2) immutable URLs = infinite CDN TTL, (3) integrity verification on retrieval, (4) saves ~30% GCS writes.
Rejected: UUID-based IDs — simple, but two identical uploads store two copies; requires cache invalidation logic on every upload; no integrity guarantee.
Cost accepted: SHA-256 computation on client before upload (~50ms on modern phones). Photo ID derived from content — slightly unusual mental model for engineers new to the system.
Reversal trigger: If Google moves to server-side hashing (unlikely — adds 162 GB/s of hashing load to upload service). SHA-256 collision not a practical concern in our lifetime.
TRADEOFF 2 — ASYNC ML vs SYNC ON UPLOAD PATH Chosen: All ML (CLIP, FaceNet, YOLO, OCR) runs async via Kafka after blob is committed. Upload completes in ~2s. ML results arrive 10–30s later.
Rejected: Sync ML on upload path. CLIP ViT-L/14 alone takes 800ms GPU inference time. Face detection adds another 500ms. Upload p99 would exceed 5s — unacceptable for a backup product.
Cost accepted: Photos unsearchable for ~30s post-upload. Users with fast fingers searching immediately after a new upload may not find it yet.
Reversal trigger: When on-device CLIP inference <100ms (feasible with Apple Neural Engine gen 5) — shift to device-side embedding, server just ingests the vector. Search becomes instant post-upload.
TRADEOFF 3 — BIGTABLE ROW KEY DESIGN Chosen: userID#reverseTimestamp#photoID. Chronological grid view = single sequential Bigtable scan — extremely efficient.
Rejected — photoID only: Uniform key distribution, but full-table scan required to get a user's library. Rejected — userID#photoID: Scan works but returns photos in ID order, not time order; requires client-side sort or secondary index overhead.
Cost accepted: Hot-row problem for power users (millions of photos concentrate load on few row ranges). Mitigated by Redis caching metadata for users active in the last 30 days.
Reversal trigger: If the primary access pattern shifts from "recent photos" to "photos of person X" — would require adding person_id as a row key dimension, which means a table migration.
TRADEOFF 4 — EAGER vs LAZY TRANSCODING Chosen: Hybrid — eager for common variants (64px, 128px, 256px, 512px, 1080p JPEG+WEBP) on upload; lazy for rare variants (4K, AVIF, custom crops) on first request.
Rejected — eager-only: Transcoding all 8+ variants costs 3× GPU time; 4K requested by <5% of users. Rejected — lazy-only: First request for any thumbnail hits 600ms+ transcode wait; catastrophic grid experience for newly-uploaded photos.
Cost accepted: Rare variant first-serve is slow (~600ms). Storage for pre-computed common variants is 5× the photo count.
Reversal trigger: If average display resolution moves to 4K (when 4K monitors become universal) → promote 4K to "eager" variant.
TRADEOFF 5 — SPANNER FOR ALBUMS vs BIGTABLE Chosen: Cloud Spanner for albums, shares, subscriptions. More expensive ($0.30/GB vs Bigtable's $0.065/GB) but provides multi-row ACID transactions.
Rejected: Bigtable for albums. Adding a photo to an album must atomically: (a) increment photo_count in albums table, (b) insert row in album_photos table. Bigtable cannot do this. With Bigtable you'd need an eventually-consistent compensating pattern, creating ghost photo counts and album corruption bugs.
Cost accepted: ~4.5× storage cost on the album dataset. Justified because album data is orders of magnitude smaller than photo metadata — the cost difference is a rounding error at system scale.
Reversal trigger: Never — this is a correctness requirement, not a performance preference.
