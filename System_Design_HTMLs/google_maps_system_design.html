Tile math

At zoom level 15 (street-level), the world is divided into 415 ≈ 1.07 billion tiles. But only ~5% cover populated areas (~53M tiles). Each vector tile is ~20KB, so the hot dataset is ~1TB — easily cacheable across CDN edge nodes. A user panning across a city viewport loads ~12 tiles per interaction, and with 1B daily users making ~10 interactions each, that's ~120B tile loads/day → ~1.4M/s average, ~20M/s at peak.

3. API design

Three independent API surfaces — each optimized for its access pattern.

GET /tiles/v3/{z}/{x}/{y}.pbf → 200 OK | Content-Type: application/x-protobuf → Cache-Control: public, max-age=604800, immutable → ETag: "v2024w15-z15-x16384-y10924"
POST /directions/v1/route { "origin": { "lat": 37.7749, "lng": -122.4194 }, "destination": { "lat": 37.3382, "lng": -121.8863 }, "mode": "driving", "alternatives": true, "depart_at": "2026-04-15T08:00:00Z" } → { "routes": [{ "polyline": "...", "duration_s": 3240, "distance_m": 77400, "steps": [...] }] }
GET /places/v1/search?q=coffee+near+me &lat=37.7749&lng=-122.4194&radius=2000 → { "results": [{ "place_id": "ChIJ...", "name": "Blue Bottle", "lat": 37.7821, "lng": -122.4093, "rating": 4.5 }] }

Why these API choices matter

Tile API uses GET with immutable cache headers — CDN can serve 95%+ of requests without hitting origin. The tile key (z/x/y) is deterministic, so cache invalidation is a simple version bump in the URL prefix. Route API uses POST because the request body can be complex (waypoints, avoid tolls, departure time). Search API uses GET because results are cacheable for short TTLs (popular queries like "gas station near [highway exit]" are highly repetitive).

Why separate services?

Tile serving is 99% cache reads — it scales horizontally by adding CDN edge nodes. Routing is CPU-bound graph computation — it scales by partitioning the world graph. Search is index-lookup heavy — it scales like a search engine (sharded inverted indexes). Coupling them would mean scaling expensive routing compute to handle tile traffic, wasting money. Each service has its own SLO, on-call rotation, and deploy cadence.

5. Deep dive — tile serving

The highest-traffic component. At ~20M req/s, this is fundamentally a caching and CDN optimization problem.

Tile addressing: the quadtree

The world map is recursively divided into 4 quadrants at each zoom level. Zoom 0 = 1 tile (the whole world). Zoom 1 = 4 tiles. Zoom z = 4z tiles. At zoom 21 (building level), there are ~4.4 trillion possible tiles, but only ~5% over populated areas are ever requested. The tile key z/x/y deterministically identifies any tile — this is what makes caching trivial.

Request flow

Three-tier caching strategy

L1 — CDN edge (95% hit rate): Tiles are immutable per version. Cache-Control: immutable with ~7-day TTL. On map data update, version in URL path bumps, old tiles naturally expire. ~200 edge PoPs globally.

L2 — Regional cache (4% hit rate): Shared cache at the data center level. Catches requests that miss at a specific CDN PoP but are popular regionally. Redis or Memcached cluster, ~10TB per region.

L3 — Origin render (~1% of requests): On a complete cache miss, the tile renderer reads raw map data (roads, buildings, labels from the map data store), generates the vector tile on demand, caches it, and returns. Rendering takes ~50ms per tile — acceptable for the 1% that reaches origin.

Pre-rendering vs on-demand

Pre-rendering all tiles would take ~1TB storage × periodic rebuilds. Instead, Google uses on-demand rendering with aggressive caching. Popular tiles (cities at common zoom levels) warm up in CDN within minutes of a version bump. Obscure tiles (rural zoom-21) render on first request and stay cached until the next version.

6. Deep dive — routing engine

The algorithmically hardest component. The world road network has ~1 billion nodes and ~2 billion edges. Naive Dijkstra would take minutes per query. Google returns routes in <100ms.

Contraction hierarchies (CH)

The core insight

During offline preprocessing, rank every node by "importance" — interstate highway intersections rank highest, residential cul-de-sacs rank lowest. Then iteratively remove low-importance nodes, adding "shortcut edges" that preserve shortest-path distances. At query time, run bidirectional Dijkstra where both searches only traverse "upward" in the hierarchy. The two search frontiers meet at a highway-level node after visiting ~1000 nodes instead of millions.

Preprocessing pipeline

Input: Raw road network (~2B edges with distance, speed limit, road class, turn restrictions).

Step 1 — Node ordering: Rank nodes by edge difference (how many shortcuts needed), number of contracted neighbors, and geographic reach. This takes ~2-4 hours on a cluster.

Step 2 — Contraction: Remove nodes in rank order, adding shortcut edges where needed. Result: the original graph + ~3B shortcut edges. Total graph size ~5B edges.

Step 3 — Partitioning: Divide the global graph into ~1000 cells using S2 geometry cells (or administrative regions). Each cell fits in memory (~1-2 GB). Cross-cell routing uses "boundary nodes" — the highway exits at cell borders.

Frequency: Full rebuild weekly (with map data updates). Incremental updates for new road segments daily.

Query-time routing

Local routes (within one cell): Single-partition CH query. Load the cell's graph into memory, run bidirectional upward-Dijkstra. ~2ms.

Cross-region routes: Identify origin cell and destination cell. Route from origin to the nearest boundary nodes, route between boundary nodes using a pre-computed overlay graph (only ~50K highway nodes globally), then route from boundary nodes to destination. Total: ~10-20ms.

With traffic: Apply a sparse "traffic overlay" — weight multipliers for congested edges (~5% of edges at any time). This means re-running CH with adjusted weights, slightly slower (~15ms) but within budget.

7. Deep dive — search & geocoding

Two entry points, same infrastructure: "coffee near me" (place search) and "123 Main Street" (geocoding).

Dual-index architecture

Text inverted index

Standard search engine architecture. Every place has a document: name, category, address, keywords. The inverted index maps tokens → posting lists of place IDs. Sharded by geography (one index shard per region) so a query for "coffee" only searches the user's region, not globally. ~200M places worldwide, ~50GB index per regional shard.

S2 spatial index

Google's S2 geometry library maps the Earth's surface to a Hilbert curve. Every lat/lng maps to an S2 cell ID at various levels. Nearby points → nearby cell IDs. A "near me" query becomes: find my S2 cell → expand to neighboring cells → retrieve all place IDs in those cells → intersect with text index results → rank.

Why S2 over geohash? S2 cells have uniform area at each level (geohash cells distort near poles), the Hilbert curve preserves locality better, and S2 supports arbitrary polygonal region queries (not just rectangles).

Ranking model

Score = f(text_relevance, distance, place_quality, personalization)

• Text relevance: BM25 on name/category match
• Distance decay: Exponential decay from user location, weighted by place category (restaurants = tight radius, hospitals = wider)
• Place quality: Rating, number of reviews, photo count, claimed business status
• Personalization: Past visits, search history, time of day (coffee shops rank higher in morning)

Geocoding path

Address parsing → structured lookup

"123 Main Street, San Jose, CA" is parsed into components: house number, street name, city, state. The lookup then does a hierarchical search: state → city → street → interpolate house number from address ranges stored per street segment. This is a different code path from place search — it's a structured tree traversal, not a ranked retrieval.

8. Traffic service

The system that makes routes dynamic rather than static. Combines real-time crowdsourced data with historical patterns.

Data collection

Real-time signals: Anonymized location pings from ~1B Android devices and Google Maps users. Each ping: lat/lng, speed, heading, timestamp. Map-matched to the nearest road segment. Aggregated per segment into a speed reading every ~60 seconds.

Historical patterns: Years of aggregated traffic data per road segment, bucketed by day-of-week × time-of-day. "Highway 101 southbound is typically 35 mph at 5:30 PM on Tuesdays."

Serving layer

Traffic tiles: Color-coded polylines (green/yellow/red) overlaid on map tiles. Generated per-segment, aggregated into regional tiles, cached with 60-second TTL.

ETA prediction: ML model combining real-time speed, historical patterns, time of day, weather, events. The model predicts per-segment travel time for the next 30-60 minutes, which the routing engine uses as edge weights.

9. Tradeoffs & alternatives

Key decisions and what you'd discuss unprompted to hit "Outstanding" on the rubric.

Vector tiles vs raster tiles

Chose vector for 10x bandwidth savings and dynamic styling. Tradeoff: requires capable GPU on client. Mitigation: serve raster as a fallback for low-end devices and embed previews. If you detect a slow client (via User-Agent or client-reported metrics), redirect to the raster tile CDN.

Contraction hierarchies vs transit node routing

Chose CH because it supports live traffic updates with moderate overhead. Transit node routing is 100x faster at query time but assumes static edge weights — rebuilding the transit node set on every traffic update is prohibitively expensive. For the rare "I need sub-1ms routing" case (embedded nav in cars), pre-compute the top-N corridor routes between major cities using transit nodes as a complement.

On-demand tile rendering vs full pre-render

Chose on-demand because 95% of tiles are never requested. Pre-rendering 1B+ tiles at every zoom level would take days and waste storage. On-demand + aggressive caching means you only pay render cost for tiles users actually view. Tradeoff: first user to view an obscure tile gets ~50ms extra latency. Mitigation: pre-warm popular tiles (top 100 cities at zoom levels 0-16) after each data update.

S2 geometry vs geohash for spatial indexing

Chose S2 for uniform cell area (geohash distorts near poles), better Hilbert-curve locality, and support for polygonal queries. Tradeoff: S2 is more complex to implement and less widely understood by the team. Mitigation: wrap S2 in a clean library interface so most engineers interact with "find places in radius" and never touch cell math directly.

Graph partitioning strategy

Chose S2 cell-based partitioning over admin-region partitioning because S2 cells have predictable, uniform sizes. Admin regions (countries, states) vary wildly — Russia is one partition while Luxembourg is another, creating massive imbalance. Tradeoff: S2 cells can split cities that straddle cell boundaries. Mitigation: allow ~10% overlap at boundaries so cross-boundary local routes don't require multi-partition queries.

10. EM-specific angles

What differentiates your answer as an EM vs an IC — weave these into your design naturally.

Staffing & team structure

"I'd organize this as 4 teams: Tile Platform (3 backend + 1 SRE), Routing Core (4 backend + 1 ML for ETA), Search & Geocoding (3 backend + 1 ML), and Traffic (2 backend + 1 data eng + 1 ML). The routing team needs the most senior engineers — contraction hierarchies is the hardest algorithm. I'd hire for graph algorithm expertise specifically."

Rollout plan

"Phased rollout for any map data or algorithm change: canary → 1% → 10% → 50% → 100%, with automated rollback on SLO breach. For routing algorithm changes specifically, I'd run shadow traffic (new algorithm alongside old, compare results) for 2 weeks before any user-facing rollout. Bad routes have real-world safety impact."

Incident readiness

"On-call rotation per team. Top 3 runbooks: (1) CDN cache purge gone wrong — fallback to L2/origin, scale origin fleet 5x temporarily. (2) Stale traffic data — disable real-time overlay, fall back to historical. (3) Routing returning absurd ETAs — circuit-break the traffic overlay, serve static-weight routes. Each has an automated detection + pager trigger."

Cross-team coordination

"This system touches the Maps Data team (raw road network), the ML Platform team (ETA model training infra), the Ads team (local business search results), and the Android/iOS platform teams (client rendering). I'd set up a shared design doc + bi-weekly sync with each adjacent team, and a quarterly roadmap alignment for the full Maps org."
