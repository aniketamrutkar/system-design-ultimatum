System Overview

We're designing a personalized news feed (Twitter/X style) where 50M daily active users read a feed of posts from people they follow, create posts, and interact with content. The defining engineering challenge is the fanout problem: when a user with millions of followers posts, how do we efficiently deliver that post to every follower's feed?

Core Design Decisions (Preview)

Hybrid Fanout (Push + Pull)

Push for normal users (<10K followers), pull on read for celebrities (>10K followers). This solves the celebrity problem without sacrificing feed freshness for 99.9% of users.

Sorted Set Feed Cache (Redis)

Each user's feed is a Redis ZSET of post IDs scored by timestamp. O(log N) insert and range query. 600GB total across a 20-node cluster.

3-Stage ML Ranking

Candidate generation → Light ranker → Heavy ranker. Balances ranking quality against latency budget (80ms for ML inference within 300ms total).

Event-Driven via Kafka

All writes produce events consumed by fanout, search indexer, analytics, and notification services. Decouples write path from downstream processing.

API Design

       Body: { content, media_ids[], reply_to? }
       Response: { post_id, created_at }

GET    /v1/feed?cursor=X&limit=20        → Get personalized feed
       Response: { posts[], next_cursor, has_more }
       ↳ cursor is opaque string encoding (timestamp, post_id)

POST   /v1/users/{id}/follow             → Follow user
DELETE /v1/users/{id}/follow             → Unfollow user

POST   /v1/posts/{id}/like               → Like a post
DELETE /v1/posts/{id}/like               → Unlike

GET    /v1/posts/{id}                    → Get single post with replies
GET    /v1/users/{id}/posts?cursor=X     → User's own posts (timeline)
GET    /v1/search?q=term&cursor=X        → Full-text search

Cursor-based vs Offset pagination: We use cursor-based pagination (encoding last-seen timestamp + post_id). Offset pagination breaks when new posts are inserted — user sees duplicates or misses posts. Cursor is stable under concurrent writes. Tradeoff: can't "jump to page 50" (not needed for infinite scroll).

Fanout Deep Dive — The Heart of Feed Design

The Celebrity Problem

This is the defining engineering challenge. Consider:

A celebrity with 10 million followers posts once. Pure push fanout means writing that post ID to 10 million Redis sorted sets. At 5K celebrity posts/sec peak, that's 50 billion cache writes/sec. This is impossible — no cache cluster survives this.

But pure pull fanout (compute feed on every read) means 150K reads/sec, each requiring fetching posts from all ~200 followees. That's 30 million DB reads/sec. Also impossible.

The solution is hybrid fanout.

Toggle: Fanout Strategies Compared

Push Only (Fanout-on-Write)

How it works: When a user posts, immediately write the post_id to every follower's feed cache.

Pros: Feed reads are O(1) — just read from cache. Extremely fast read path.

Cons: Explodes with celebrities. User with 10M followers = 10M writes per post. Fanout latency can reach minutes. Massive Redis write throughput required. Wasted writes for inactive followers (70% may never read that day).

Math: 5K celebrity posts/sec × 1M avg celebrity followers = 5 billion writes/sec. Need ~5,000 Redis nodes just for fanout. Cost: $2M+/month.

Verdict: REJECTED — doesn't scale for celebrity users

Pull Only (Fanout-on-Read)

How it works: No precomputed feeds. On every feed read, fetch the user's followee list, then query each followee's recent posts, merge and rank.

Pros: Write path is trivial — just persist the post. No celebrity problem. Always fresh.

Cons: Read path is expensive. 200 followees × "get latest 10 posts" = 2,000 DB reads per feed request. At 150K feed reads/sec = 300M DB reads/sec. Read latency: 200-500ms even with caching.

Math: Even with aggressive caching (90% hit), still need 30M DB reads/sec. Requires massive read replica fleet.

Verdict: VIABLE but slow — used by some systems but latency suffers

Hybrid Push/Pull ✓ (Our Approach)

How it works: Split users by follower count. Normal users (<10K followers): push fanout-on-write. Celebrities (>10K followers): pull-on-read at feed assembly time.

Push path (99.9% of users): Avg 200 followers → 200 Redis ZADD per post. At 280 non-celebrity posts/sec: 56K Redis writes/sec. Very manageable.

Pull path (0.1% of users, ~30% of content): Feed Service knows which followees are celebrities. On feed read, parallel-fetch their latest posts (cached in Redis, hit rate >95%). Adds ~15ms to read path.

Why 10K threshold? Below 10K, fanout completes in <100ms and total cache write load is sustainable. Above 10K, fanout latency grows linearly and dominates total write throughput. 10K is the knee of the curve — empirically determined.

Verdict: CHOSEN — best of both worlds

Fanout Worker Architecture

Implementation Details

Fanout workers are Kafka consumers processing posts.created events.

Consumer Group: fanout-workers (32 consumers, 32 partitions)

For each PostCreated event:
  1. Check is_celebrity flag on author
     → If celebrity: SKIP (handled on read path). Done.
     → If normal user: continue

  2. Fetch follower list from Social Graph cache (Redis SET)
     → Returns list of follower_ids

  3. For each follower_id:
     a. ZADD feed:{follower_id} {timestamp} {post_id}
     b. ZREMRANGEBYRANK feed:{follower_id} 0 -801
        (trim to 800 entries)

  4. Commit Kafka offset after all Redis writes succeed

Parallelism: Each worker fans out to followers in batches of 100, using Redis pipeline (50x less round-trips than individual commands).

Ordering guarantee: Kafka partition key = author_id, so all posts from the same author are processed in order within a partition. Cross-author ordering is not guaranteed but not needed — ranking handles final ordering.

Staffing: Fanout service owned by the Feed Infrastructure team (3 engineers). Key hire: someone with deep Kafka + Redis operational experience. Monitoring: Primary metric is "fanout lag" — histogram of time between PostCreated event timestamp and last follower cache update. Dashboard: per-partition consumer lag, Redis write latency p99, throughput. Alert: p99 fanout lag > 30s → PagerDuty. > 5 min → auto-scale consumers from warm pool.

Feature Store

Architecture

The Feature Store is a Redis-backed precomputed feature cache. Features are computed offline (batch) and in near-real-time (streaming).

User Features (precomputed hourly):
  user:{id}:features → {
    avg_daily_sessions: 3.2,
    preferred_content_types: [image, video],
    active_hours: [9,10,11,12,18,19,20,21],
    topic_affinities: {politics: 0.3, tech: 0.8, sports: 0.5},
    engagement_rate: 0.04
  }

Post Features (computed at creation + updated streaming):
  post:{id}:features → {
    author_reputation: 0.85,
    has_media: true,
    media_type: "image",
    early_engagement_velocity: 15.2,  // likes/min in first 10min
    content_quality_score: 0.72,
    hashtag_trending_score: 0.4
  }

Cross Features (precomputed daily):
  affinity:{user_id}:{author_id} → {
    interaction_count_30d: 12,
    profile_visit_count: 3,
    reply_rate: 0.15
  }

Latency: Feature lookup via Redis MGET: <5ms for 50 posts' features. Cache hit rate: 99%+ (features are precomputed and always warm).

Model complexity vs latency: We could use a larger model (GPT-scale) for ranking and get ~5% better engagement. But inference would take 500ms+, blowing our latency budget. The 3-stage pipeline is the engineering compromise: cheap filter, expensive model on small candidate set. This is exactly what Google/Meta/Twitter do in production.

A/B testing discipline: Every ranking model change goes through: (1) Offline evaluation on historical data (AUC, NDCG), (2) Shadow mode (run in parallel, log predictions, don't serve), (3) A/B test on 1% of users for 7 days, (4) Measure: DAU retention, time spent, interaction rate, (5) Full rollout only if statistically significant improvement (p < 0.05). Team: 3 ML engineers + 1 data scientist. Weekly ranking review meeting with product.

Data Store Deep Dives

Every data store choice is justified by its specific workload characteristics.

1. Redis Cluster — Feed Cache
AspectDetail
Why RedisSub-ms latency + native Sorted Set (ZSET) data structure. ZREVRANGEBYSCORE gives paginated chronological feed in O(log N + M). No other store offers this combo.
Why not MemcachedMemcached lacks sorted sets. We'd serialize/deserialize entire feed list on every read/write — O(N) instead of O(log N). At 800 items/feed, ~10x slower for pagination.
Schemafeed:{user_id} → ZSET(post_id, timestamp)
post:{post_id} → Hash(text, author, media_urls, counters)
user:{id}:follows → SET(followee_ids)
Size math50M users × 800 post IDs × 16 bytes = ~600GB. 20-node cluster (32GB RAM each) with replication.
Read throughput~200K ops/sec (ZREVRANGE + MGET for hydration)
Write throughput~60K ops/sec (ZADD from fanout workers)
EvictionLRU on inactive users (no login 7 days). Cold start penalty: ~2s feed reconstruction.
Failure modeAuto-failover to replica (<5s). During failover, 5% of users get cold-start. No data loss — Redis is cache, source of truth is PostgreSQL.
2. PostgreSQL (Sharded) — Posts Source of Truth
AspectDetail
Why PostgresPosts need ACID transactions, strong consistency, and proven durability. Fixed schema, standard query patterns → relational is ideal. MVCC for concurrent reads.
Why not Cassandra/DynamoDBPosts have a fixed schema with known query patterns (by post_id, by author_id+time). We don't need Cassandra's write-optimized LSM or multi-DC replication at this scale. Postgres is simpler to operate and offers richer indexing.
Schemaposts(post_id BIGINT PK, author_id BIGINT, content TEXT, media_urls JSONB, created_at TIMESTAMPTZ, like_count INT, reply_count INT)
Indexes(author_id, created_at DESC) for timeline, (post_id) for direct lookup. Snowflake ID embeds shard routing.
Sharding64 shards by author_id % 64. "All posts by user X" hits single shard. Post lookups use shard routing from Snowflake ID.
Per-shard size~400K posts/day → ~140M posts/year → ~420M rows at 3-year retention. Well within Postgres capability with monthly table partitioning.
Replication1 primary + 2 read replicas per shard. Sync replication to 1 replica (RPO=0). Feed hydration reads go to replicas.
Failure modePatroni auto-failover to sync replica (<30s). Zero data loss. During failover, writes return 503; clients retry. Reads continue on remaining replica.
3. Social Graph Store — TAO-style (MySQL + Redis)
AspectDetail
Why not Neo4jOur queries are simple adjacency lookups ("who does X follow?", "who follows X?"), not multi-hop graph traversals. A sharded relational store + cache is simpler, cheaper, and faster for these patterns. Neo4j enterprise licensing at 10B edges is extremely expensive.
Data modelfollows(follower_id, followee_id, created_at). Two copies: forward index (shard by follower_id), reverse index (shard by followee_id).
Why two copies"Who do I follow?" shards by follower_id. "Who follows me?" shards by followee_id. Single-shard queries in both directions. Tradeoff: 2x storage, write amplification on follow/unfollow.
CachingRedis SETs for follower/followee lists. 95%+ cache hit rate. TTL=24h with event-driven invalidation on follow/unfollow.
ConsistencyRead-after-write on primary shard. Feed sees eventual consistency (up to 60s) — acceptable UX.
4. Kafka — Event Bus
AspectDetail
Why KafkaDurable, ordered, replayable event log. Multiple consumers (fanout, search, analytics, notifications) each read independently. At 500K msgs/sec, Kafka is proven at this scale (LinkedIn runs 7T msgs/day).
Why not RabbitMQ/SQSRabbitMQ: doesn't scale to 500K msgs/sec without massive clustering overhead. SQS: no ordering guarantees, no consumer group semantics, higher latency. Kafka's log-based model is ideal for event sourcing.
Topicsposts.created (32 partitions), interactions (16 partitions), follows (8 partitions)
Cluster12 brokers, replication factor 3, ISR min 2
Retention7 days (allows replay for backfill or recovery)
5. Elasticsearch — Search Index
AspectDetail
Why ESFull-text search with relevance scoring + near-real-time indexing (<1s). 25M new docs/day is standard ES workload. Supports trend detection via aggregations.
Index designPosts index: 10 shards, 1 replica. Fields: text (analyzed), author_id (keyword), created_at (date), hashtags (keyword[]), engagement_score (float).
IndexingKafka consumer → ES bulk API (batches of 500, every 1s). Lag <2s from post creation to searchability.

Failure Modes & Graceful Degradation

CAP Theorem Position

We choose AP (Availability + Partition tolerance) for the feed read path. During network partitions, feeds may show stale data but the service remains available. The post write path is CP — we'd rather reject a write than lose a post.

This is the correct tradeoff: a slightly stale feed is acceptable, a lost post is not.

Cascading Failure Prevention

Circuit Breakers

Every inter-service call has a circuit breaker (Hystrix-style). If error rate > 5% over 30s window, circuit opens → requests go to fallback path immediately instead of waiting for timeouts. Half-open state tests recovery every 10s.

Bulkheads

Thread pools are isolated per downstream dependency. If the ML ranking service is slow, it can't exhaust threads needed for Redis reads. Each dependency gets its own connection pool with hard limits.

Load Shedding

When feed service CPU > 80%, start shedding by priority: anonymous feeds first, then search, then authenticated feeds (highest priority). Return 429 with Retry-After header.

Timeout Budgets

Total request timeout = 500ms. Each step gets a fraction. If Redis takes 100ms (normally 5ms), remaining steps get reduced budgets. Prevents slow components from causing total timeout.

Consistency vs Availability on Unfollow: When user A unfollows user B, B's posts should disappear from A's feed. We could synchronously purge B's posts from A's feed cache (consistent but slow) or let them age out naturally over 1-24 hours (fast but inconsistent). We choose eventual purge via background job: Kafka event triggers async cache update. Worst case, user sees 1-2 posts from unfollowed user before purge completes. This is what Twitter does.

Rollout Strategy

Phased Rollout Plan

Phase 1 (Week 1-2): Shadow mode — New feed path runs in parallel with existing. Responses logged but not served. Compare latency, correctness, and engagement prediction accuracy.

Phase 2 (Week 3): Internal dogfood — Serve to employees only (~1K users). Catch obvious bugs and UX issues.

Phase 3 (Week 4): 1% canary — Random 1% of users. Monitor: error rate, p99 latency, engagement metrics (likes/session, time-on-feed). Kill switch: auto-rollback if error rate > 0.5% or p99 > 500ms.

Phase 4 (Week 5): 10% rollout — If canary is clean for 7 days. Expand monitoring to business metrics (DAU, ad revenue per session).

Phase 5 (Week 6-7): 50% → 100% — Gradual ramp. Keep old path warm for 2 weeks as rollback target.

Scale Reasoning — Every Decision Grounded in Numbers

Traffic Derivation

Given: 50M DAU

Read Path:
  Avg feed reads/user/day: 10 (open app, scroll, refresh)
  Total reads/day: 50M × 10 = 500M
  Avg reads/sec: 500M / 86400 = ~5,800/sec
  Peak multiplier: 25-30x (lunch hour + evening surge in largest timezone)
  Peak reads/sec: ~150,000/sec

Write Path:
  Avg posts/user/day: 0.5 (not all users post daily)
  Total posts/day: 50M × 0.5 = 25M
  Avg writes/sec: 25M / 86400 = ~290/sec
  Peak multiplier: 15-20x
  Peak writes/sec: ~5,000/sec

Read:Write ratio: 30:1 — classic read-heavy workload

Storage Derivation

Posts:
  25M posts/day × 1KB avg (text + metadata, excl. media) = 25GB/day
  Per year: ~9TB
  3-year retention: ~27TB across 64 shards = ~420GB per shard
  → Well within PostgreSQL per-instance capacity

Feed Cache (Redis):
  50M users × 800 post IDs × (8 byte post_id + 8 byte score) = 640GB
  With Redis overhead (~2x): ~1.2TB total
  20 nodes × 64GB RAM (50% utilization) = 640GB usable ✓

Post Cache (Redis):
  Hot posts: ~100M active posts × 1KB = 100GB
  5 nodes × 32GB RAM

Social Graph:
  10B edges × 24 bytes (two int64 IDs + timestamp) = 240GB
  2x for forward + reverse index = 480GB
  16 MySQL shards (30GB each)
  Redis cache for hot adjacency lists: ~50GB

Media (S3):
  ~30% of posts have images: 7.5M images/day × 500KB avg = 3.75TB/day
  Per year: ~1.4PB
  S3 tiering: hot (30 days) → infrequent access → glacier

Why Each Store Was Chosen (Scale-Justified)

DecisionScale FactorWhy This, Not That
Redis ZSET for feed cache 200K ops/sec, <1ms At 150K reads/sec, we need sub-ms latency. Only Redis gives sorted set with O(log N) operations at this speed. Memcached can't sort. PostgreSQL can't serve at this latency.
PostgreSQL for posts (not Cassandra) 5K writes/sec 5K writes/sec is easily within Postgres capability (single node handles 50K+). We don't need Cassandra's write-optimized LSM at this write volume. Postgres gives us richer queries, ACID, and simpler ops.
Kafka (not SQS/RabbitMQ) 500K msgs/sec, 5 consumer groups Multiple independent consumers (fanout, search, analytics, notifications) each reading the same events. Kafka's consumer group model is purpose-built for this. SQS would need 5 separate queues with message duplication.
64 PostgreSQL shards 25M posts/day, 3-year retention 27TB / 64 = 420GB per shard. Postgres handles this with monthly partitioning. At 10x scale (500M DAU), we'd reshard to 256+ shards — but cross that bridge when we get there.
Hybrid fanout threshold at 10K Balance: 56K writes/sec (push) + 15ms read overhead (pull) Below 10K: push fanout is 200 writes per post, total ~56K/sec — manageable. Above 10K: each post would add 10K+ writes. At 5K celebrity posts/sec, that's 50M+ writes/sec — Redis cluster can't handle it.
20-node Redis cluster 600GB data, 200K ops/sec Each node: 32GB RAM, 10K ops/sec. 20 nodes give 640GB capacity and 200K ops/sec throughput. Auto-failover with replicas. At 10x scale: 200 nodes (still manageable with Redis Cluster protocol).

What Changes at 500M DAU (10x)

At 10x scale, several design decisions need to change:

Feed Cache: 200-node Redis cluster becomes operationally painful. Consider migrating to a custom distributed cache (like Twitter's Manhattan) or partitioned Memcached with a sorted-set abstraction layer.

PostgreSQL sharding: 256+ shards with Vitess or CockroachDB for automated resharding. Cross-shard queries become a real problem — may need to precompute more aggregations.

Fanout: Celebrity threshold may need to drop to 5K. Consider geographic fanout partitioning (EU users processed by EU fanout workers).

Multi-region: At 500M DAU, single-region latency is unacceptable for global users. Need active-active multi-region with conflict resolution for the social graph and posts.
