Request lifecycle — rider books a trip
  • 1
    Rider taps "Request ride"
    App sends pickup GPS + destination to API Gateway
    JWT validated, rate limit checked, request forwarded to Trip service. Idempotency key generated. Fare estimate computed from route + surge cache.
  • 2
    Pricing service computes fare estimate
    Base + distance + time + surge multiplier from Redis
    Surge multiplier is read from Redis cache (pre-computed every 60s per H3 cell). ETA computed via road network + live traffic. Auth hold placed on payment method.
  • 3
    Matching engine runs batch optimisation
    500ms batch window, Hungarian algorithm
    Open requests accumulate for 500ms. Engine queries Redis GEO for nearby drivers, scores each on ETA (50pts) + rating (15pts) + acceptance rate (15pts) + idle time (10pts) + heading (10pts). Joint optimisation minimises total ETA across all riders in batch.
  • 4
    Offer dispatched to top-ranked driver
    WebSocket push with 15-second timeout
    Offer sent over existing persistent WebSocket. Driver has 15s to accept or decline. On decline/timeout, next-ranked driver receives offer. Acceptance rate tracked rolling window.
  • 5
    Trip created — downstream events fire
    Kafka publishes trip.matched event to all consumers
    Trip state transitions to ACCEPTED in PostgreSQL. Kafka emits to: notification service (push to rider), location tracking begins, payment hold confirmed, safety monitoring starts.

Why batch instead of instant match?
Greedy (instant)
  • Each rider gets closest driver immediately
  • Locally optimal per rider
  • Globally suboptimal — can leave nearby riders stranded
Batch (optimal)
  • Wait 500ms, optimise all at once
  • Minimises sum of all ETAs
  • Some riders wait 200ms more — system 20–40% faster

Trip state machine
REQUESTED → MATCHED → ACCEPTED → EN_ROUTE → ARRIVED → IN_PROGRESS → COMPLETED
Each transition fires a Kafka event. PostgreSQL enforces valid transitions via check constraints at the DB level — no application bug can corrupt state. Redis caches the live state for sub-ms reads by the matching engine and driver app.

GPS update pipeline
1
Driver app sends GPS every 4 seconds
Fused Location Provider (GPS + cell + Wi-Fi) over persistent WebSocket
The OS fused location provider combines GPS chip, cell tower triangulation, and Wi-Fi positioning. When the driver is stationary (detected via accelerometer), GPS polling pauses entirely to save battery. Each frame carries: driver_id, lat, lng, bearing, speed_kmh, accuracy_m.
2
WebSocket server → Kafka location-updates topic
100k msg/s published; partitioned by driver_id
WebSocket servers are stateless — driver connections are pinned via consistent hashing so all messages from one driver go to the same server instance. Kafka partitioning by driver_id ensures ordered delivery. The Kafka step decouples ingestion from storage so a slow Redis write never blocks the driver app.
3
Location consumer deduplicates and writes to Redis GEO
GEOADD overwrites previous position; sub-ms
The consumer checks if the position has changed by more than 5m before writing — suppressing GPS jitter for stationary drivers. GEOADD atomically updates the driver's position in the sorted set. A companion hash stores driver status (AVAILABLE / ON_TRIP) and pre-computed ETA.
4
Filtered positions pushed to rider app via WebSocket
Only drivers within ~5km radius; bearing included for smooth animation
The server filters by the rider's S2 cell — only nearby drivers are sent. Each frame includes bearing so the car icon rotates correctly. The rider app interpolates movement between received frames using dead reckoning (extrapolate from speed + heading) so dots glide rather than teleport every 4 seconds.
5
Raw stream also consumed by TimescaleDB
Source of truth for fare calculation — distance + time from GPS trace
The raw GPS trace is the source of truth for the final fare — not the pre-trip route estimate. After trip completion, the fare service queries TimescaleDB for all positions during the trip, computes actual km and minutes, and uses those figures to calculate the final charge. TimescaleDB hypertable partitioned by time; old partitions auto-compressed.
S2 cells vs Geohash
Geohash
  • Base32 string — prefix = proximity
  • Simple — built into Redis natively
  • Edge distortion at cell boundaries
S2 cells (Google)
  • 64-bit integer — Hilbert curve ordering
  • Numerically close = spatially close
  • Better edge properties; used by Uber

Payment lifecycle — 5 stages
1
Auth hold placed at booking
Estimated fare + 20% buffer frozen on card — no money moves yet
The payment gateway is called with an authorization (not a capture) for estimated fare × 1.2. The card network reserves the amount. If the card declines, the platform tries the next saved payment method. Auth holds expire after 7 days if never captured.
2
Trip runs — no payment activity
Meter accumulates from GPS trace in TimescaleDB
The fare meter runs server-side from the GPS trace, not a client timer. Distance and time are derived from raw positions. The auth hold remains frozen. No payment API calls during the trip.
3
Final fare computed on trip completion
Base + (km × rate) + (min × rate) × surge − promos + tax
The fare service queries the GPS trace from TimescaleDB, computes actual distance and duration, applies surge and any promo codes, adds tolls as pass-through. The final amount is always ≤ the auth hold (if a detour pushed it higher, a new auth is needed).
4
Capture request sent with idempotency key
Gateway converts auth to actual charge for exact final fare
The idempotency key (UUID derived from trip_id + "capture") is written to Redis before the gateway call. Any retry with the same key returns the original result — preventing double-charge on network failures. Capture completes in <2s after trip ends.
5
Nightly batch settles driver payout
Platform retains 17–25% commission; driver gets 75–80%
A nightly batch job aggregates all completed trips per driver, computes net payout after commission and tax, and initiates ACH/NEFT transfers. Suspended drivers' earnings are held. Failed payouts retry the next night; 3 consecutive failures trigger manual review. "Instant Pay" (push-to-card) available as paid feature.

How the multiplier is computed
  • 1
    Count demand per cell
    Open requests + near-term forecast based on historical patterns
    Not just current open requests — the system forecasts demand for the next few minutes using historical curves. A concert ending in 5 minutes will spike demand before riders even open the app.
  • 2
    Count supply per cell
    Available drivers + drivers finishing trips nearby (forward supply)
    Forward supply includes drivers expected to complete trips in adjacent cells within the next few minutes. This prevents over-surging a zone that will have ample supply in 2 minutes.
  • 3
    Compute demand/supply ratio
    Ratio → nonlinear multiplier curve
    Below 0.7× ratio: stay at 1×. 0.7–1.5× ratio: gentle ramp. Above 1.5×: steeper curve to send strong price signal. The curve shape is empirically calibrated — the weakest surge that reliably attracts enough drivers.
  • 4
    Dampen and hold the signal
    Ramp up gradually; hold for 3–5 min minimum
    Raw ratio is noisy. A 2-second demand burst shouldn't show riders a 3× surge that vanishes. The multiplier ramps gradually. Once activated, it's held for a minimum window so drivers who see it and start repositioning can trust it'll still be there when they arrive.
  • 5
    Blend adjacent cells
    Smooth gradient — no hard cliff at cell boundary
    A concert zone's surge radiates outward. Adjacent cells are blended using a weighted average so a driver 1.5 cells away sees an elevated (smaller) multiplier — still worth repositioning toward the zone.
  • 6
    Write to Redis cache; serve to rider
    Rounded to discrete steps — 1.2×, 1.5×, 2.0× etc.
    The multiplier is stored per H3 cell in Redis with a 90-second TTL. The fare estimate endpoint reads from Redis — never recomputes demand/supply live. The rider sees rounded discrete steps rather than "1.873×" to reduce cancellation friction.

The most underrated surge trigger
Rain. Demand spikes 40% (nobody walks to the bus) while supply drops 20% (some drivers don't like driving in rain, and traffic slows per-trip earnings). Combined effect creates the worst supply gaps. Platforms with weather API integration pre-arm surge and driver bonuses as soon as a heavy rain forecast appears — before the gap manifests.

Why async over synchronous calls?
Synchronous chain
  • Trip svc calls 6 downstream services
  • Total latency = sum of all latencies
  • One slow svc stalls the whole chain
  • Adding a consumer = modify trip svc
Async event bus
  • Trip svc publishes one event, returns
  • Consumers run in parallel, independently
  • Slow consumer only lags itself
  • New consumer: zero changes to producer

Durability: consumer offsets and replay
Every consumer tracks its own read position (offset) independently in Kafka. If the fraud detection service crashes and restarts, it resumes exactly from the last committed offset — replaying every missed event. No data is lost. No manual reconciliation. This is the critical difference from RabbitMQ, which deletes messages on consumption.
ORDERING GUARANTEE
All events for the same trip_id go to the same partition. payment.captured always processes after trip.completed.
AT-LEAST-ONCE
Default delivery guarantee. Consumers must be idempotent. Exactly-once available via transactional producers for critical flows.
SCHEMA REGISTRY
All events serialised with Avro + schema registry. Adding a field is backward-compatible. Removing requires version bump.

Ops & Safety
// supply gap response playbook + safety feature architecture
Supply gap response — work tiers in order
Tier 1 — Automatic (0–2 min) · Zero cost
  • Surge rises — price signal pulls idle drivers online (takes 3–5 min to show effect)
  • Match radius expands — offers sent to drivers up to 8km away (vs 3km default)
  • Forward dispatch — drivers finishing nearby trips pre-matched into gap zone
Tier 2 — Ops-triggered (2–8 min) · Low cost
  • Push to offline drivers — targets drivers offline <2hr, within 5km (still nearby)
  • Zone bonus — ₹50–100/trip for 30 min window (direct budget cost)
  • Redirect from surplus zones — widen offer radius in adjacent areas
Tier 3 — Escalation (8+ min) · High cost / risk
  • Emergency incentive blast — city-wide; trains drivers to wait for blasts
  • Rider expectation management — in-app message reduces cancellations
  • Engineering escalation — persistent gap despite Tier 1+2 = structural problem

Rollout strategy — every component
  • 1
    Feature flags first
    No change goes to production without a flag
    LaunchDarkly or internal flag service. Every new feature starts OFF. Rollout is a config change, not a deployment. Instant rollback without any code change.
  • 2
    Canary deployment via API Gateway
    1% → 5% → 20% → 100% with automated gates
    The API Gateway splits traffic by weight between stable and canary versions. Automated gates compare error rate, p99 latency, and business metrics between cohorts. Gate fails → instant rollback to 0% canary, no human needed at 3am.
  • 3
    Shadow mode for critical services
    Run new version alongside old — compare outputs
    For matching engine changes: run the new algorithm in shadow mode, comparing its assignments against the live engine. No rider or driver is affected. Statistical significance testing before any traffic switch.
  • 4
    Database migrations zero-downtime
    Expand → migrate → contract pattern
    Adding a column: deploy code that writes both old and new column (expand). Run migration to backfill. Deploy code that reads new column only. Drop old column (contract). Never lock the table. Never require downtime.

P0 — Matching down city-wide
Page matching team lead + SRE + EM immediately. War room in 5 min. All other P2+ pages suppressed during P0.
P1 — Payment capture failure rate >5%
Page payments on-call. 15-min response SLA. Check gateway status page first — often third-party issue.
