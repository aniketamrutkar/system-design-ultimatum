Requirements & Clarifications

Always clarify before designing. The problem says "slowest queries" — you need to know what that means precisely.

Clarifying questions to ask the interviewer
  • Definition of "slow": Absolute threshold (>500ms) or relative (top 1% by latency)?
  • Freshness: Real-time (last 5 min) or last 24 hours? Dashboard refresh rate?
  • Granularity: Per-query-pattern or per individual query instance?
  • Debug context: What data helps engineers debug? Query plan? Shard breakdown? Network hops?
  • Scale: Starting with one server — what's the eventual target? (They'll say thousands.)
  • Internal only: Authenticated? Yes — this is an internal engineering tool.

High-Level Architecture

The pipeline has four conceptually distinct tiers: instrumentation (inside search servers), transport (Kafka), aggregation (stream processing), and serving (dashboard API + stores).

Single-server mode (Phase 1)
One search server writes directly to a local file / sidecar process that does the same pattern fingerprinting and percentile computation locally. Dashboard reads from a local store. No Kafka required. This is the MVP that proves correctness before adding distribution complexity.
Scale-out trigger (Phase 2+)
The moment you add a second server, you need a centralized aggregation point — because P99 across servers is not the average of per-server P99s. This is the key distributed systems insight: percentiles don't compose naively. You must merge the raw distributions.

Component Deep Dives

Two sub-problems deserve the most depth: the sampling strategy and the percentile computation algorithm. These are where most candidates hand-wave, and where interviewers push hardest.

1. Query fingerprinting — why it matters

Without fingerprinting, every unique query is a separate entry. "WHERE id=1", "WHERE id=2", "WHERE id=1000" are three separate rows in your monitoring system — useless. You need to normalize:

// Raw queries (useless for grouping) "SELECT * FROM docs WHERE user_id=12345 ORDER BY rank LIMIT 10" "SELECT * FROM docs WHERE user_id=99999 ORDER BY rank LIMIT 10" // Fingerprinted template (groupable) "SELECT * FROM docs WHERE user_id=? ORDER BY rank LIMIT ?" // → hash: 0x7fa3b21c ← this is your partition key and group-by // Implementation: replace numeric literals, quoted strings, and // list constants with placeholder tokens. Same approach as // pg_stat_statements in PostgreSQL.

2. Why T-Digest for percentiles — not averages, not reservoir sampling

This is the most technically rich part of the design. You cannot store every query latency and sort them. You need a sketch — a compact data structure that approximates percentiles.

Approach
Pros
Cons
Exact sort
100% accurate
O(n log n) time, O(n) space — impossible at 85M events/pattern/day
Simple average
O(1) time, O(1) space
Completely hides tail latency. P99=5s looks like P50=50ms in the average. Useless for slow query detection.
Reservoir sampling
Fixed memory
Loses precision at tails — exactly where you care most. If 1% of queries are slow, reservoir underrepresents them.
HdrHistogram
Good tail precision
Fixed range — must declare max value upfront. Can't merge across servers for different configs.
T-Digest ✓
Exact at extremes (P1, P99), sub-percent error mid-range, mergeable across nodes, streaming-friendly
Slightly more complex than HdrHistogram; not in all standard libraries

The mergeability is critical: each Flink worker has a T-Digest for its subset of queries. Merging two T-Digests is O(n) where n is the number of centroids (typically ~300). This is how you get global P99 across 10K servers — merge 10K T-Digests, not 10K sorted lists.

3. Tail-biased sampling — always capturing what matters

// Sampling decision (runs after query completes, off critical path) func shouldSample(latency_ms int64) bool { // Always sample slow queries (potential bugs, regressions) if latency_ms > currentP95Threshold { return true // 100% capture } // Sample 1% of normal queries (baseline health signal) return rand.Float64() < 0.01 } // currentP95Threshold: fetched from Redis, refreshed every 60s // Starts at 500ms for a new server; adapts as Flink computes real P95 // Fail-open: if Redis unreachable, default to 1000ms threshold

The threshold adapts. Initially you hardcode 500ms. Once the system has enough data (after a few minutes), Flink pushes the current P95 back to Redis and the search servers pull it. This means the definition of "slow" evolves with the system's actual performance profile.

Data Store Justification

Each store is chosen for a specific access pattern. The worst design is picking one store for everything.

Hot Redis Sorted Set
Real-time top-K slow patterns
Key: top-slow-queries:{dc}
Score: P99 latency (ms)
Member: pattern_hash

Why Redis ZADD: O(log n) insert, O(k) top-K retrieval. Dashboard reads ZREVRANGE to get current worst patterns instantly. Updated every minute by Flink.

Alternative rejected: Bigtable scan for top-K — would require full table scan on each dashboard load. Redis is 100× faster for this access pattern.
Warm Bigtable
7-day percentile history per pattern
Row key: {pattern_hash}#{minute_bucket}
Columns: p50, p95, p99, count, sample_trace_ids
TTL: 7 days

Why Bigtable: Range scans by time work naturally with the row key design. Column families separate stats from trace samples. Handles 10K writes/sec from Flink with ease.

Alternative rejected: Elasticsearch — full-text indexing overhead for a structured time-series use case. Bigtable is simpler and 5× cheaper.
Cold BigQuery
Full trace storage for debug drill-down
Schema: Parquet columns — trace_id, query_text, latency_ms, server_id, shard_ids, query_plan_json, timestamp
Retention: 30 days, then archive to GCS

Why BigQuery: Ad-hoc queries on full traces (e.g., "show me all traces for pattern X where shard 17 was slow"). Columnar storage makes latency-filter queries fast.

Alternative rejected: Storing full traces in Bigtable — each trace is 5-50KB. At 85M traces/day even at 1% sampling, that's 850K * 50KB = 40GB/day — too expensive for Bigtable's pricing model.
Transport Kafka
Durable trace event stream
Topic: query-traces
Partitions: 256 (scaled with server count)
Partition key: query_fingerprint_hash
Retention: 24 hours (reprocessing window)

Why hash-based partitioning: Same query pattern always goes to same Flink worker. This is critical — T-Digest state for a pattern must live in exactly one place or you need a merge step.

At-least-once delivery: Flink consumers commit offsets after processing. Duplicate events add noise to percentile sketches but don't cause correctness issues.

Failure Modes & Mitigations

A monitoring system that goes down when the thing being monitored is in trouble is useless. Design failures explicitly.

F1 Kafka goes down
▶
Impact
Trace events are lost. Dashboard goes stale. Search serving is unaffected.
Mitigation: Search servers have an in-memory ring buffer (10K events, 5s flush). If Kafka is down for <5s, events buffer. For longer outages, we fail open — drop the trace, emit a metric (kafka_producer_drop_total), alert the monitoring team. Search latency is NOT increased. Dashboard shows stale data with a banner: "Data delayed — Kafka ingestion disrupted."
F2 Flink lag / crash
▶
Impact
Dashboard percentiles are stale. Real-time top-K in Redis stops updating. Alerts may fire late.
Mitigation: Kafka retains events for 24 hours. Flink restores from checkpoints (saved to GCS every 60s). On recovery, Flink replays from the last checkpoint — we get eventual consistency within minutes. Window state is persisted so T-Digest sketches are not lost. The monitoring of the monitoring: a secondary heartbeat process checks that Flink is consuming and alerts if lag > 10 minutes.
F3 Redis evicts data under memory pressure
▶
Impact
Top-K sorted set loses entries. Dashboard shows fewer or stale slow query patterns.
Mitigation: Redis eviction policy set to allkeys-lru (evict least-recently-used keys). The top-K sorted set is small (100 members × 200 bytes = 20KB) and access frequency is high — it will not be evicted. For safety, Redis is sized at 4× expected working set. If it is evicted, Flink rebuilds it within one window period (1 minute) from Bigtable.
F4 Hot partition in Kafka
▶
Impact
A single query pattern is queried 100× more than others — its hash always hits the same Kafka partition, creating a hot partition that overwhelms one Flink worker.
Mitigation: Detect hot patterns using partition lag metrics. For hot patterns, add a suffix-based fan-out: partition key becomes {pattern_hash}#{random_suffix_0-9} to spread across 10 partitions. Flink then has a two-stage reduce: per-shard T-Digest → merge step. This is handled transparently if detected during traffic analysis.
F5 Sampling biases the percentiles
▶
Impact
With tail-biased sampling (100% slow, 1% fast), the sample is not representative of the full distribution. P50 computed from 1% of fast queries may be wrong.
Mitigation: Weight samples inversely by sampling rate when computing percentiles. A trace sampled at 1% counts as 100 events; a trace sampled at 100% counts as 1 event. This is standard practice in distributed tracing (Zipkin, Jaeger all do this). The result is an unbiased estimate of the full distribution from a biased sample.

EM Lens — Beyond the Technical Design

A Staff engineer can design this. An EM candidate needs to show they're thinking about staffing, rollout, org impact, and operational readiness.

Staffing & Team Composition
I'd staff this project with 4 engineers over 3 months:

2 backend engineers — instrumentation library (search server side) and Flink aggregation pipeline. These need strong distributed systems knowledge. The instrumentation work is particularly tricky because it touches the search serving critical path — needs engineers who can reason about lock-free data structures and async I/O.

1 data infrastructure engineer — Kafka cluster sizing, Bigtable schema design, BigQuery pipeline. Ideally someone with prior Bigtable or GCS experience.

1 frontend engineer — Dashboard UI. Needs to understand time-series visualization and latency histograms. This role can start 3-4 weeks in once the API contract is stable.

Critical path: Instrumentation → Kafka → Flink can be parallelized from day 1. Dashboard development can start against a mock API. Data store setup should be done by week 2.
Rollout Strategy
Phase 1 (Week 1-3): Single-server MVP. Instrument one search server. Local aggregation. Basic dashboard with manual refresh. Goal: prove the fingerprinting and percentile logic is correct against known slow queries.

Phase 2 (Week 4-6): Add Kafka and Flink. Instrument 10 servers (1% of fleet). Dark launch — run pipeline in parallel, compare against Phase 1 output to validate correctness. No user-facing change yet.

Phase 3 (Week 7-8): Roll out instrumentation to 100% of fleet. Enable dashboard for the search infrastructure team (internal). Collect feedback on what debug context is missing.

Phase 4 (Week 9-12): Add alerting, cross-DC aggregation, historical drill-down. Expose to broader engineering org.

Rollback plan: The instrumentation library has a kill switch (feature flag) that disables all trace collection with a config push. Flink pipeline shutdown does not affect search serving.
Incident Handling & On-Call
On-call responsibilities: The monitoring pipeline itself needs an on-call rotation. Monitoring of the monitoring: a separate canary process checks that dashboard freshness is within 10 minutes and pages if not. This is separate from the search on-call.

Top 3 runbooks to write at launch:
1. Flink lag > 10 min — check Kafka consumer lag, check Flink job manager logs, restart from checkpoint
2. Redis top-K stale > 5 min — check Redis memory, verify Flink ZADD is succeeding
3. P99 spike alert — false positive check: is it correlated with a deployment? With a Kafka lag event? Only page search on-call after ruling these out.

SLO for the monitoring system itself: Dashboard freshness <15 min, 99.5% of the time. Alert latency <5 min for P99 spikes. These are internal tooling SLOs — not as tight as search serving.
Cross-functional Alignment
This system crosses Search Infrastructure (owns search servers), Data Infrastructure (owns Kafka/Bigtable/BigQuery), and Reliability Engineering (owns alerting/on-call).

At kickoff: shared design doc reviewed by all three teams. Weekly sync for first 6 weeks. Explicit API contracts between the instrumentation library (owned by Search Infra) and the aggregation pipeline (owned by Data Infra).

The instrumentation library will be contributed back to a shared observability platform — it shouldn't be a search-specific one-off. This creates a coalition of supporters and avoids duplication if other teams (e.g., Ads, Maps) want the same capability.

Why T-Digest fits in a single Flink worker's memory
Each T-Digest sketch uses ~300 centroids × 16 bytes = ~5KB per pattern. With 3,500 active patterns per worker, that's 3,500 × 5KB = 17.5 MB of sketch state per worker. A typical Flink task manager has 4GB heap — sketch state is 0.4% of available memory. This is why T-Digest is the right choice: it's compact enough to hold the state for all patterns in a single worker's heap.
