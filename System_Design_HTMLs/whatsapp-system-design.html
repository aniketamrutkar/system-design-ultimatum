Component Deep Dives

CRITICAL WebSocket Gateway

▸

What It Does

Maintains long-lived, persistent WebSocket connections with every online client. This is the real-time nervous system — every message, receipt, typing indicator, and presence update flows through here. At our scale, this is 25M concurrent connections across ~50 server instances.

Why It Exists as a Separate Service

Decoupling connection management from business logic is essential. WS connections are stateful and long-lived (hours/days), while business logic services need to be stateless and horizontally scalable. If you couple them, deploying a new version of your chat logic tears down millions of connections and causes a reconnection storm.

Key Technical Decisions

Language: Go or Rust. We choose Go for this layer. Each connection needs ~30KB of memory (buffers + goroutine stack). At 500K connections per box, that's ~15GB RAM per server — well within 64GB machines, leaving room for headroom and OS overhead. Rust would give us lower per-connection overhead (~10KB) but Go's goroutine model maps perfectly to the "one connection = one lightweight thread" pattern, and the ecosystem for WebSocket handling is mature.

Connection-to-Server Mapping: We store the mapping of user_id → ws_server_id in Redis. When the Chat Service needs to deliver a message to user B, it looks up which WS server holds B's connection and sends it there via internal gRPC. This lookup is O(1) in Redis and adds ~1ms latency.

Session stickiness via consistent hashing: The L4 load balancer uses consistent hashing on user_id to route reconnections to the same WS server when possible. This reduces the window of "connection migration" during reconnects. But we never depend on stickiness — the Redis session store is the source of truth.

Connection Lifecycle

// Simplified connection handler func handleConnection(ws *websocket.Conn, userID string) { // 1. Register in session store redis.Set("session:{userID}", serverID, TTL=30m) // 2. Heartbeat loop (client sends ping every 30s) // Server extends TTL on each ping // If no ping for 60s → connection dead → cleanup // 3. Read pump: client messages → Kafka producer go readPump(ws, kafkaProducer) // 4. Write pump: internal gRPC delivery → client go writePump(ws, deliveryChan) }

Data Structures

Per-connection buffer ring: Each connection has a 64KB write buffer ring. Messages queue here if the client's TCP window is full (slow network). If the ring fills, we start dropping low-priority messages (typing indicators first, then presence, never chat messages). Chat messages go to the offline queue instead.

Failure Mode

WS Server Crash

If a WS server dies, 500K connections drop simultaneously. Clients detect this within 30-60 seconds (heartbeat timeout) and reconnect. The L4 LB re-routes them to surviving servers. During the reconnect window, the Sync Service queues messages for these users. The blast radius is 500K/25M = 2% of online users, and recovery is fully automatic.

Graceful Drain

For planned maintenance, we drain connections gradually: send a "reconnect" control frame to 5% of connections per second. Clients reconnect to a new server. Full drain takes ~20 seconds. Zero message loss.

EM Perspective

Staffing: 2 senior engineers own this service. It's the most operationally sensitive component — on-call rotation is critical. Deploy cadence: Weekly at most, always during low-traffic windows (2-4 AM local), with connection draining. Key metric: Connection count per server, reconnection rate (spikes indicate problems), message delivery latency histogram.

CORE Chat Service — Message Routing & Ordering

▸

What It Does

The Chat Service is the brain of message routing. It receives messages from the WS Gateway, validates them, persists them to Cassandra, determines which recipients need the message, and dispatches delivery to the correct WS servers (or the Push/Offline queue for offline users).

Why It Exists

Separating routing logic from connection management means we can scale, deploy, and reason about message handling independently. The Chat Service is stateless — any instance can handle any message — enabling trivial horizontal scaling.

Message Ordering — The Hard Problem

Users expect messages in a conversation to appear in order. But in a distributed system, "order" is ambiguous. We use a Hybrid Logical Clock (HLC) approach:

  • Client-side timestamp (physical clock) for rough ordering
  • Server-assigned sequence number per conversation for strict ordering within a conversation
  • The sequence number is generated atomically using a Redis INCR on key seq:{conversation_id}

This gives us a total order within each conversation, which is all users care about. We do NOT need global ordering across conversations.

// Message processing pipeline func processMessage(msg Message) { // 1. Validate: size, rate limit, conversation membership validate(msg) // 2. Assign sequence number (atomic) msg.SeqNum = redis.Incr("seq:" + msg.ConvID) // 3. Persist to Cassandra (async, but before ACK) cassandra.Write(msg) // CL=LOCAL_QUORUM // 4. ACK to sender ("sent" checkmark) sendAck(msg.SenderID, msg.MsgID, SENT) // 5. Route to recipient(s) for _, recipient := range getRecipients(msg.ConvID) { if isOnline(recipient) { deliverViaWS(recipient, msg) } else { queueOffline(recipient, msg) sendPush(recipient, msg) } } }

Key Tradeoff: ACK Before or After Persist?

We ACK to the sender after Cassandra write confirms at LOCAL_QUORUM. This adds ~5ms latency but guarantees that a "sent" checkmark means the message is durable. The alternative (ACK immediately, persist async) risks message loss if the Chat Service crashes between ACK and persist. For a messaging product, durability > latency.

Idempotency

Clients may retry sending the same message (network timeout after send, before ACK). We use the client-generated msg_id (UUID) as an idempotency key. Before processing, we check if msg_id already exists in Cassandra. If so, we return the existing ACK. This check is fast because msg_id is part of the Cassandra partition key.

Chat Service Instance Crash Mid-Processing

If the service crashes after persist but before delivery: the message is safe in Cassandra. The recipient will pick it up on their next sync (or another delivery attempt from the Kafka consumer, which retries undelivered messages).

HIGH SCALE Group Service — Fan-out Strategy

▸

The Fan-out Problem

When a user sends a message to a 500-member group, we need to deliver it to 499 other users. At 350K messages/sec total, if even 10% are group messages with avg 50 members, that's 35K × 50 = 1.75M delivery operations/sec just for groups. The fan-out strategy is the most important design decision here.

Approach: Hybrid Fan-out (Write + Read)

Small groups (≤50 members): Fan-out on Write

When a message arrives, immediately push to each member's delivery channel. This is eager delivery — higher write amplification but lower read latency. Users get messages pushed to them instantly.

Large groups (51-1024 members): Fan-out on Read

Write the message once to the group's timeline. When a member opens the group, they pull the latest messages. This avoids the thundering herd problem of pushing to 1000 members simultaneously, most of whom may have the group muted anyway.

The threshold of 50 is calibrated based on our scale numbers: 50-member fan-out at 35K msgs/sec = 1.75M deliveries/sec, which our WS layer handles comfortably. But 500-member fan-out would be 17.5M deliveries/sec, which would saturate our WS gateway's internal gRPC capacity.

Group Membership Storage

Stored in PostgreSQL with a denormalized cache in Redis (SET per group). Membership changes are infrequent (adds/removes) compared to messages, so a write-through cache is appropriate. Cache invalidation on membership change is synchronous — we cannot deliver messages to stale member lists.

EM Perspective — Phased Rollout

Start with fan-out-on-write for all groups (simpler). Instrument delivery latency and WS gateway CPU. When large group delivery latency exceeds SLO, implement the hybrid approach. This avoids premature optimization and lets production data drive the threshold.

SECURITY End-to-End Encryption (Signal Protocol)

▸

Why Signal Protocol

Signal Protocol (used by WhatsApp, Signal, and Facebook Messenger's secret mode) provides: forward secrecy (compromising a key doesn't decrypt past messages), future secrecy (self-healing after compromise), and deniability (messages can't be cryptographically proven to have been sent by a specific party).

Server's Role: Key Distribution Only

The server NEVER sees plaintext. It stores and distributes public key bundles (pre-keys) for offline users. The encryption/decryption happens entirely on the client device. This is architecturally significant because it means:

  • Server-side message search is impossible (by design)
  • Message backup requires client-side key backup (encrypted with user passphrase)
  • Group messages use the Sender Key distribution protocol (one encrypt, each member decrypts)

Key Server

Stores pre-key bundles in PostgreSQL. Each user uploads 100 one-time pre-keys. When another user initiates a session, one pre-key is consumed (atomically deleted). When pre-keys run low (<20), the server pushes a "replenish keys" request to the client. This is a write-heavy workload for popular users (lots of people initiating chats with them), which is why we use PostgreSQL with row-level locking on the pre-key table.

Impact on Architecture

E2E encryption constrains several other design decisions: no server-side message indexing/search, media is encrypted client-side before upload (so CDN serves opaque blobs), and message backup to cloud requires a separate user-held backup key. These are acceptable tradeoffs for privacy guarantees.

INFRA Presence & Typing Indicators

▸

Design Philosophy: Optimistic, Lossy, Cheap

Presence and typing indicators are ephemeral signals. If a user misses a "typing" indicator, nothing bad happens. This lets us relax consistency and durability requirements dramatically compared to chat messages.

Implementation

Online/Offline Presence: Stored in Redis with TTL. When a user connects, set presence:{user_id} = ONLINE with 60s TTL. The heartbeat (every 30s) refreshes the TTL. When TTL expires, user is implicitly offline. No explicit "offline" event needed — this handles crashes, network drops, and app kills gracefully.

Last Seen: Updated in Redis on disconnect (or TTL expiry). Persisted to PostgreSQL asynchronously (batch write every 5 minutes) so it survives Redis restarts.

Typing Indicators: Pure fire-and-forget UDP-style. Client sends "typing" to WS Gateway, which relays to the other participant's WS server directly (no Kafka, no persistence). If the relay fails, we silently drop it. Typing indicators have a client-side 5-second auto-expiry.

Scale Trick: Presence Fan-out Filtering

We do NOT broadcast presence changes to all contacts. Instead, the client subscribes to presence for the conversations currently visible on screen (typically 10-15 chats). The server maintains a reverse index in Redis: presence_subscribers:{user_id} → [subscriber_ids]. When user A's presence changes, we only notify users who have actively subscribed. This reduces presence fan-out by ~95%.

MEDIA Media Service — Upload, Transcode, CDN

▸

Upload Flow (Pre-signed URL Pattern)

Client requests an upload URL from Media Service → Media Service generates a pre-signed S3 PUT URL (valid 15 minutes) → Client uploads directly to S3, bypassing our servers entirely. This is critical at 6K uploads/sec — routing 100MB files through our application servers would saturate network bandwidth instantly.

Processing Pipeline

After upload completes, S3 fires an event to our processing queue (SQS/Kafka):

  • Images: Generate thumbnails (150px, 300px, 600px). Thumbnails are sent inline with the message notification so the recipient sees a blurred preview immediately.
  • Video: Transcode to H.264 at multiple bitrates (240p, 480p, 720p) for adaptive streaming. Generate a poster frame.
  • Documents: Generate preview thumbnail of first page.

All media is encrypted client-side before upload. The server processes encrypted blobs — it cannot see the content. Thumbnails are also encrypted (generated client-side before upload as a separate blob).

Download via CDN

Media is served through a CDN (CloudFront/Akamai). Since media is encrypted, CDN caching works normally — the CDN serves opaque encrypted blobs. Decryption happens on the recipient's device. CDN cache hit rate target: >80% for media accessed within 24 hours.

Storage Economics

At 50TB/day, storage costs dominate. We use S3 Intelligent Tiering: media accessed within 30 days stays in Standard, then auto-moves to Infrequent Access, then Glacier Deep Archive after 1 year. WhatsApp-style "media auto-delete after 30 days on server" further bounds storage growth.

RELIABILITY Sync Service — Offline Message Queue

▸

Problem Statement

When a user is offline for hours or days, messages accumulate. When they come back online, we need to deliver all missed messages quickly and in order, without overwhelming the client or the network.

Architecture

Each user has a sync cursor — a (conversation_id, sequence_number) pair that tracks the last message they received in each conversation. This cursor is stored in Redis (hot) and PostgreSQL (durable).

On reconnect, the client sends its sync cursor. The Sync Service queries Cassandra for all messages after the cursor, groups them by conversation, and streams them to the client in reverse chronological order (newest conversations first, newest messages first within each conversation). This matches the UX — users see their most recent conversations immediately.

Throttling & Backpressure

If a user has 10K unread messages, we don't dump them all at once. We use progressive loading: send the 20 most recent messages per conversation for the 50 most recent conversations (= 1000 messages max in first batch), then let the client request more via pagination. This keeps the initial sync under 2 seconds.

Compaction

For very long offline periods, the offline queue can grow large. We compact: if a user has 500 messages in one group, we send only the latest 50 and a "X more messages" indicator. The client fetches older messages on scroll. This bounds the sync payload regardless of offline duration.

Cassandra Schema — Messages Table

▸

Table Design

CREATE TABLE messages ( conversation_id UUID, bucket INT, -- time bucket (day number) seq_num BIGINT, -- from Redis INCR msg_id UUID, sender_id UUID, encrypted_body BLOB, -- E2E encrypted msg_type TINYINT, -- text/image/video/etc media_url TEXT, created_at TIMESTAMP, PRIMARY KEY ((conversation_id, bucket), seq_num) ) WITH CLUSTERING ORDER BY (seq_num DESC);

Why This Schema

Partition key = (conversation_id, bucket): The bucket (day number since epoch) prevents unbounded partition growth. Without bucketing, a conversation with 1M messages over 5 years would be a single giant partition — terrible for Cassandra performance. With daily buckets, each partition holds at most one day's messages (~200-500 per active conversation).

Clustering key = seq_num DESC: Most reads are "give me the latest N messages" — descending order means the newest messages are read first in a single sequential disk read. No sorting needed.

Why not timestamp as clustering key? Two messages could have the same timestamp (clock precision). The sequence number is guaranteed unique and ordered within a conversation.

Read Pattern

-- "Latest 50 messages in conversation X" SELECT * FROM messages WHERE conversation_id = 'X' AND bucket = current_bucket LIMIT 50; -- If fewer than 50 in today's bucket, query previous bucket(s) -- Client-side merge of results across buckets

Compaction Strategy

We use TWCS (TimeWindowCompactionStrategy) with a window of 1 day, matching our bucket size. This is optimal for time-series data because it avoids compacting old data with new data, reducing write amplification. Old buckets are never modified (messages are immutable), so they compact once and stay compacted.

TTL and Data Lifecycle

Messages have a configurable TTL (default: never expire for 1:1, 90 days for groups). When a user deletes a message, we write a tombstone. Cassandra's gc_grace_seconds is set to 10 days to allow all replicas to receive the tombstone before purging.

Redis — Session & Presence Schema

▸

Key Design

// Session: which WS server holds this user's connection "sess:{user_id}" → "{ws_server_id}:{connection_id}" TTL=60s // Presence: is user online? "pres:{user_id}" → "online" TTL=60s // Last seen (for offline users) "last:{user_id}" → 1713196800 (unix timestamp, no TTL) // Conversation sequence counter "seq:{conversation_id}" → 284519 (monotonic, no TTL) // Group membership cache "grp:{group_id}" → SET{user_id1, user_id2, ...} TTL=1h // Presence subscribers (who's watching user X's presence) "psub:{user_id}" → SET{watcher1, watcher2, ...} TTL=5m

Why Redis Cluster (not Memcached, not DynamoDB)

vs Memcached: We need data structures (SETs for group membership, INCR for sequence numbers, TTL for presence). Memcached is pure key-value with no data structure support. Redis gives us these primitives natively without application-level serialization/deserialization overhead.

vs DynamoDB: DynamoDB can handle the throughput but adds 3-10ms latency per operation. Redis is sub-millisecond. On the message delivery hot path, every millisecond matters — the session lookup is in the critical path for every single message delivery.

Durability Trade-off

Redis is in-memory with AOF persistence. If a Redis node dies, we lose recent session data. This is acceptable because: sessions are re-established on client reconnect (within seconds), presence is inherently ephemeral, and sequence numbers are also tracked in Cassandra (last seq_num per conversation). The only risk is a brief period of duplicate sequence numbers during Redis recovery, which we handle with idempotency checks.

Failure Modes & Tradeoffs

CAP Theorem Position

▸

Our system is fundamentally AP with tunable consistency. Here's how this manifests per component:

ComponentCAP ChoiceRationale
Message Store (Cassandra)AP with LOCAL_QUORUMAvailability over strict consistency. LOCAL_QUORUM gives us "strong enough" consistency — 2 of 3 replicas must confirm. In a network partition, the partition with 2+ replicas continues serving; the minority partition rejects writes. This is the right tradeoff: users prefer sending messages with a small risk of temporary inconsistency over being unable to send at all.
Session Store (Redis)APIf session lookup fails, the message goes to offline queue. User gets it on next sync. No data loss, minor delay.
User/Group DB (PostgreSQL)CPProfile and membership changes need strong consistency. It's acceptable for group member updates to briefly make the service unavailable (user sees "retry") rather than deliver messages to a stale member list (security issue — removed member receiving messages).
Pre-key Store (PostgreSQL)CPPre-key consumption MUST be strongly consistent. If two users consume the same pre-key, the encryption session is broken. ACID transaction required.

Failure Scenarios & Graceful Degradation

▸
Cassandra Node Failure (1 of 3 replicas)

With RF=3 and LOCAL_QUORUM, losing 1 node has zero user impact. Reads and writes continue at 2/3 quorum. The failed node catches up via Cassandra's built-in anti-entropy repair on recovery.

Mitigation

Monitor node count per datacenter. Alert if <2 nodes available for any token range. Auto-replace failed nodes within 1 hour via infrastructure automation.

Kafka Broker Failure

Kafka partition leader fails. Follower is elected as new leader (takes 1-5 seconds). During this window, messages in-flight to that partition will timeout and retry. With acks=all and min.insync.replicas=2, no message loss occurs. Client-side retries absorb the brief disruption.

Mitigation

Run 5+ brokers per cell with RF=3. Use rack-aware placement to survive full rack failures. Monitor under-replicated partitions as a leading indicator.

Redis Cluster Partition

If Redis cluster splits, sequence number generation may produce gaps or (in rare cases) duplicates. Session lookups may fail, routing messages to offline queue unnecessarily.

Mitigation

Use Redis Cluster with 6 nodes (3 primary + 3 replica) across 3 AZs. Automatic failover within 15 seconds. For sequence numbers, accept gaps (they don't affect ordering, just aren't contiguous). Idempotency at Cassandra layer catches any duplicate sequences.

Entire Cell Failure (Region Outage)

All 5M users in the cell lose service. This is the worst case. DNS-based failover to the nearest cell takes 30-60 seconds (TTL). During this window, messages queue at the sender's client.

Mitigation

Cell assignment has 10% headroom — each cell can absorb a neighboring cell's users temporarily. Cross-cell Cassandra replication (async, EACH_QUORUM on select critical data) enables read-your-writes on failover. Practice cell evacuation drills quarterly.

Rollout & Canary Strategy

▸

Deployment Pipeline

Every service follows the same 4-stage deployment pipeline:

  1. Shadow (Day 0): Deploy new version alongside production. Mirror 1% of traffic. Compare response correctness and latency against production. Zero user impact.
  2. Canary Cell (Day 1): Deploy to our smallest cell (São Paulo, 2M users). Monitor for 24 hours. Key metrics: message delivery success rate, P99 latency, error rate, crash rate.
  3. Regional Rollout (Day 2-3): Deploy to 3 more cells (Mumbai, Singapore, London). Monitor 24 hours each. If any cell shows regression, halt and rollback.
  4. Global (Day 4-5): Deploy to remaining cells. At this point we have 48+ hours of production data on 4 cells.

Rollback

Every deployment creates a rollback artifact. Rollback is a single command that: reverts the container image, drains connections (for WS Gateway), and is automated if error rate exceeds 0.1% or P99 latency exceeds 500ms. Target: rollback completes in <5 minutes.

Feature Flags

All new features are behind feature flags (LaunchDarkly or similar). This decouples deployment from feature launch. We can deploy code to all cells, then gradually enable the feature for 1% → 10% → 50% → 100% of users. If the feature causes issues, we disable the flag instantly without a code deploy.

Scale Reasoning — From Numbers to Decisions

Starting from First Principles

1. Message Throughput → Cassandra

Input: 10B messages/day ÷ 86400 seconds = ~115K msgs/sec average, ~350K peak (3x factor).

Per write: avg 200 bytes payload + ~100 bytes metadata = ~300 bytes per message. With RF=3, each write becomes 3 physical writes = ~1M physical writes/sec at peak.

Why Cassandra: A single Cassandra node handles ~10K-15K writes/sec sustained. At 1M physical writes/sec, we need ~70-100 nodes. This is well within Cassandra's proven operating range (Apple runs 150K+ nodes, Netflix runs 10K+ nodes). PostgreSQL with sharding could theoretically work, but managing 100+ shards with consistent hashing, rebalancing, and cross-shard queries is exactly what Cassandra was built to eliminate.

Cost at this scale: 100 nodes × c5.4xlarge (16 vCPU, 32GB RAM) = ~$45K/month. DynamoDB at 1M WCU = ~$190K/month. Cassandra wins 4x on cost.

2. Connection Count → WebSocket Server Fleet

Input: 50M DAU × 50% concurrent = 25M WebSocket connections.

Per connection: ~30KB memory (8KB read buffer + 8KB write buffer + 4KB goroutine stack + 10KB connection metadata). 30KB × 500K = ~15GB per server.

Fleet size: 25M ÷ 500K = 50 servers. Using c5.2xlarge (8 vCPU, 16GB RAM) would be tight; we use c5.4xlarge (16 vCPU, 32GB) for headroom and CPU for TLS termination.

Why not HTTP/2 Server Push? HTTP/2 multiplexes over a single TCP connection but still requires a full HTTP request/response cycle per message. WebSocket is a framed protocol — after the initial handshake, each message is just a few bytes of framing. At 350K msgs/sec, the framing overhead difference is significant.

3. Session Lookups → Redis

Input: Every message delivery requires 1 session lookup (recipient → WS server). At 350K msgs/sec, that's 350K Redis GETs/sec. For group messages with avg 20 members, add 20 lookups per group message. Worst case: ~1M Redis ops/sec.

Why Redis: A single Redis node handles ~100K ops/sec. A 10-node cluster handles 1M+. Latency is <1ms P99. DynamoDB would add 3-10ms per lookup, adding 3-10ms to every message's delivery latency — unacceptable when our P99 target is 200ms total.

Memory footprint: 50M session entries × ~100 bytes each = ~5GB. Easily fits in a small Redis cluster.

4. Media Storage → S3 with CDN

Input: 5% of messages include media. 10B × 5% = 500M media files/day. Average size 100KB (heavily skewed — most are compressed images, few are large videos). 500M × 100KB = ~50TB/day.

Why S3 + CDN, not our own storage: At 50TB/day, we'd need to manage 18PB/year of storage. S3's cost is ~$0.023/GB/month for standard, dropping to $0.004/GB for Glacier. Self-managing this much storage (HDFS, Ceph) requires a dedicated storage team of 5-10 engineers. S3 is operationally free. The CDN ensures that popular media (shared in large groups, viral content) is served from edge without hitting S3 repeatedly.

Pre-signed URL pattern: At 6K uploads/sec, if each upload went through our application servers, we'd need ~60 servers just for proxying media (assuming 100MB uploads at 100Mbps per server). Pre-signed URLs eliminate this — clients upload directly to S3.

5. Cell Architecture → Blast Radius Control

Input: 50M DAU ÷ 10 cells = 5M users per cell. Each cell is a complete, independent stack.

Why cells: A bad deploy or infrastructure failure in one cell affects 5M users (10%) instead of 50M (100%). At our scale, the probability of a catastrophic failure is non-trivial — cells reduce the expected impact by 10x. The cost is cross-cell routing complexity for messages between users in different cells (~15% of all messages). This adds 20-50ms latency for cross-cell messages (inter-region network hop) which is acceptable.

Cell sizing: Each cell must be small enough that a neighboring cell can absorb its users during failover (10% headroom). 5M users is calibrated so that a cell can temporarily handle 5.5M users without performance degradation.
